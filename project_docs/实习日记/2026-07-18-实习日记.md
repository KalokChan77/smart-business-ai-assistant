3. **关键问题与解决方法**  
   - **Dify 502 不是 Key 问题**：配置已写入且权限正确，但 Nginx 仍把请求转发到旧 API 容器地址。通过对照当前容器地址和 Nginx 日志确认根因，重启 Nginx 后三类接口恢复。  
   - **8000% 置信度**：后端契约已是 0–100，前端又乘了一次 100。统一改为 `formatPercent` 直接展示，并补充回归测试。  
   - **文案与页面不一致**：页面写了“可查看客服最终确认回复”，但原先只有列表没有详情入口。补齐“查看详情”，复用现有公开工单详情接口，只在确认后展示最终回复。

## 二、当天实习工作的思考与收获

1. **技术知识**  
   进一步理解了 Vue3 路由元数据与 RBAC 动态导航、SSE `message_end` 完成态、FastAPI 与 Dify 适配边界，以及前端工具链中 TypeScript 与 `vue-tsc` 的兼容约束。也学会了在 Docker 环境下区分“Key 未配置”和“网关上游地址过期”两类故障。

2. **工程方法**  
   今天最深的体会是：**接口测试通过不等于真实交互正确**。8000% 和“看不到最终回复”都是在浏览器对照页面文案、后端契约和实际点击路径后才发现的。答辩材料同样必须对照代码，不能写入尚未实现的订单工具、自动备用模型或外部发送能力。另一个收获是安全清理：临时租户、随机密码文件和浏览器会话必须在验收后立刻删除，文档只写删除计数和“已验证”。

3. **不足与改进**  
   固定演示账号初始化脚本和一键启动说明已经补齐；后续仍要把 Dify Nginx 上游变化导致 502 的排查步骤整理成更独立的运维章节。百分比和状态枚举这类契约，以后应在类型定义、展示函数和 API 清单三处同时约束，避免再次出现重复换算。
