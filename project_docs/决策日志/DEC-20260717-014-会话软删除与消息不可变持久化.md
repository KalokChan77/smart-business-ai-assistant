- 关联迁移：`208539977b2b_create_conversations_and_messages.py`
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

技术方案把会话和消息列为核心数据模型。普通模型、LangGraph Agent 和 Dify 流程最终都需要把用户问题与完整助手回复保存到同一个会话中，并允许前端查询历史记录。该数据还会与后续 AI Run、反馈和审计记录关联，因此不能只把聊天记录临时保存在浏览器或模型服务中。

## 二、需要解决的问题

本阶段需要同时确定以下边界：

1. 用户删除会话时是否直接物理删除数据；
2. 消息是否允许原地修改；
3. 如何保证消息在相同时间戳和并发追加情况下仍保持真实顺序；
4. 如何防止其他租户或同租户其他用户通过会话 ID 读取消息；
5. `Message` 是否重复保存 `tenant_id`；
6. 会话模块如何与未来 AI Gateway 解耦。

初版实现按 `created_at + UUID` 排序，并在消息表重复保存 `tenant_id`。代码审查后发现，这会带来租户字段不一致、同时间戳顺序不可靠、并发追加导致 `last_message_at` 回退，以及仓储层消息查询边界不足等风险，因此在阶段验收前进行了修正。

## 三、可选方案

### 方案 A：物理删除会话和消息

- 优点：实现简单，数据表不会保留已删除记录。
- 缺点：无法恢复，会破坏未来 AI Run、反馈和审计记录的关联。

### 方案 B：会话和消息都允许任意修改

- 优点：用户可以直接编辑历史内容。
- 缺点：历史消息不能再作为可靠的模型输入输出证据，修改规则和审计成本较高。

### 方案 C：会话软删除，消息追加后不可变，并使用会话内单调位置编号

- 优点：界面可以隐藏已删除会话，同时保留历史关联；消息只追加不修改；位置编号能够表达真实追加顺序。
- 缺点：所有查询必须统一排除 `deleted_at`；并发追加需要事务和行锁；后续还要制定数据保留与彻底清理策略。

## 四、最终决策

- 会话使用 UUID 主键，保存 `tenant_id`、`user_id`、标题、最后消息时间、下一消息位置和 `deleted_at`；
- 普通删除接口只设置 `deleted_at`，列表、详情、消息查询和消息追加均排除已删除会话；
- 会话 API 不接受客户端提交 `tenant_id` 或 `user_id`，始终使用当前认证主体；
- 访问其他租户或同租户其他用户的会话统一返回 404，避免暴露资源是否存在；
- `Message` 不重复保存 `tenant_id`，租户和所有者通过所属 `Conversation` 推导，消除两个租户字段不一致的可能；
- 每个会话维护 `next_message_position`，消息保存正整数 `position`；数据库对 `(conversation_id, position)` 建立唯一约束，并检查 `position > 0`；
- 消息角色使用稳定小写值 `user`、`assistant`、`system`、`tool`，并由数据库 CHECK 约束保护；
- 消息按 `position` 升序查询，不再使用时间戳和 UUID 推测追加顺序；
- 追加消息时，仓储使用 `SELECT ... FOR UPDATE` 锁定目标会话，在同一事务内分配位置、递增计数器、更新 `last_message_at` 并插入消息；
- 消息仓储查询直接连接 `Conversation`，强制过滤 `tenant_id`、`user_id` 和 `deleted_at IS NULL`，不只依赖服务层先查询；
- 第一阶段列表继续使用 `limit`、`offset` 和 `total`。教学规模下契约简单，未来大数据量时再迁移到基于时间、位置和 UUID 的游标分页；
- SQL 查询集中在 `app/conversations/repository.py`，业务规则集中在 `app/conversations/service.py`。未来 AI Gateway 只调用会话服务，不直接依赖 SQLAlchemy。

## 五、决策理由

软删除兼顾用户体验和审计需要，消息不可变保证模型输入输出证据不会被原地篡改。把租户归属集中在会话表中可以避免冗余字段漂移；把所有者和软删除过滤直接写入消息仓储，可以形成纵深防御，即使未来某个服务调用遗漏前置检查，底层查询也不会返回越权数据。

`position` 是会话内明确的业务顺序，比相同时间戳下按 UUID 排序更可靠。行锁和单事务更新则避免两个并发请求读取同一个计数器、生成重复位置或让较晚提交的事务把 `last_message_at` 更新为较早时间。

## 六、影响与风险

- 正面影响：会话历史、AI 流程和审计边界统一；并发追加顺序可验证；租户字段不再重复；后续流式对话可直接复用持久化服务。
- 潜在风险：软删除数据会持续增长；行锁使同一个会话内的并发写入串行化；`offset` 分页在数据变化或数据量很大时可能重复、遗漏或性能下降。
- 当前权衡：教学与测试规模下，同一会话写入量有限，串行分配消息位置可接受；列表继续保留简单 offset 分页。
- 后续方案：达到较大数据量时新增归档/清理策略和游标分页，并另写决策日志，不在当前阶段提前增加复杂度。

## 九、可用于实习日记的素材

我认识到“删除一条会话”不仅是执行 SQL `DELETE`。AI 项目还要保留运行记录、反馈和审计关系，因此界面中的删除更适合使用软删除。代码审查还让我发现，按时间戳和 UUID 排序看似稳定，却不等于真实的消息追加顺序。通过为每个会话维护单调位置并在事务中使用行锁，可以把业务顺序转化为数据库能够验证的约束。仓储层直接执行租户、用户和软删除过滤，也让我理解了权限控制不能只依赖某一层，而应在关键数据边界形成纵深防御。
