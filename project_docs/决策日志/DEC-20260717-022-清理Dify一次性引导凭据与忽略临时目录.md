# DEC-20260717-022：清理 Dify 一次性引导凭据与忽略临时目录

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Dify 配置收尾、安全扫描、临时凭据清理
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

Dify 初始化、应用发布和三类 Key 配置已经完成，正式运行配置由受控环境变量提供。收尾安全检查发现，早期自动化引导目录仍保存了登录响应、Cookie、CSRF Token、应用 Token、Dataset 标识和插件凭据响应等一次性中间文件。虽然目录权限较严格，部分文件权限仍为普通本地可读，而且目录尚未被项目根 `.gitignore` 明确排除。

这些文件已经不再被后端、Dify Compose 或验证脚本引用，继续保留只会扩大误提交、备份扩散和本地泄露风险。

## 二、需要解决的问题

1. 配置完成后是否继续保留一次性引导响应和 Token 文件；
2. 如何避免后续工具再次生成同名临时目录并被误纳入项目文件；
3. 如何记录清理结果而不读取、输出或复制任何凭据内容；
4. 如何保留可重复部署能力，同时不依赖历史敏感中间文件。

## 三、可选方案

### 方案 A：原样保留临时目录

- 优点：可以回看早期接口响应。
- 缺点：长期保存不再需要的 Token、Cookie 和凭据响应，安全收益为零，泄露面持续存在。

### 方案 B：只修改文件权限，不删除

- 优点：操作简单，仍能保留全部历史文件。
- 缺点：权限不能阻止误备份、误上传或未来被其他工具读取；也不能满足最小化保存原则。

### 方案 C：删除凭据型引导目录，并把临时目录加入忽略规则

- 优点：从源头移除不再需要的敏感副本；后续重新生成时也不会被误提交；正式配置和运行状态不受影响。
- 缺点：无法再依赖历史响应文件复现初始化过程，后续应使用脱敏脚本输出和决策日志作为证据。

## 四、最终决策

采用方案 C：

1. 在确认后端、Dify Compose 和验证脚本均不引用一次性引导目录后，删除整个凭据型引导目录，不逐个读取或复制文件内容；
2. 在项目根 `.gitignore` 中加入 Dify 引导临时目录和插件打包临时目录；
3. 正式服务继续只从受控 `.env`、Dify 数据库和容器配置读取所需配置；
4. 后续自动化脚本只输出 `configured`、HTTP 状态、任务状态和数量，不把 Key、Token、Cookie、内部 ID 或完整上游响应落盘；
5. 如需重新初始化，应重新生成一次性凭据，验证后立即清理，不复用历史 Token 文件。

## 五、决策理由

凭据最安全的副本是不存在的副本。一次性引导文件在配置完成后已经失去业务价值，删除它们比单纯修改权限更符合最小化保存原则。可重复部署应依赖脚本、环境变量模板、决策日志和脱敏验证结果，而不是依赖一批包含真实会话状态的历史响应文件。

## 六、影响与风险

- 正面影响：减少 Token、Cookie、内部 ID 和凭据响应的本地副本；降低误提交和误备份风险；项目根忽略规则更加完整。
- 潜在风险：无法再回看早期初始化原始响应；如果正式配置本身丢失，需要在 Dify 中重新生成对应 Key。
- 控制措施：三类 Dify Key 已通过真实调用验证；正式配置文件仍保留且不进入版本控制；配置指南和决策日志记录了不含密钥的重建步骤。
- 回退方案：不恢复旧临时文件；需要时按配置指南重新生成新的临时凭据，并在验证完成后再次清理。

## 七、验证证据

- 使用文件名和引用扫描确认后端、Compose 与验证脚本不依赖一次性引导目录；
- 清理前只统计文件数量、权限和潜在凭据型文件名，没有读取或输出文件正文；
- 引导目录清理后路径不存在；
- `.gitignore` 已明确排除 Dify 引导与插件临时目录；
- 清理前后三类 Dify Key、知识查询和文档生命周期真实冒烟均已通过，正式运行不依赖该目录。

## 九、可用于实习日记的素材

这次清理让我认识到，安全不仅是把 Key 放进 `.env`，还包括管理初始化过程产生的 Cookie、Token 和响应文件。临时文件完成任务后如果长期保留，就会成为新的敏感信息副本。正确做法是确认运行不再依赖后及时删除，并通过忽略规则和自动清理避免再次积累。
