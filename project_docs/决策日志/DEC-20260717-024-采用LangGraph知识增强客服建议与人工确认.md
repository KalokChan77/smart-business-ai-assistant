# DEC-20260717-024：采用 LangGraph 知识增强客服建议与人工确认

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：客服工单、问题分类、知识增强建议回复、人工编辑确认
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

需求基线 P4 要求完成“知识库上传、检索、引用与客服辅助”，核心交付范围还要求实现“客服推荐回复和人工确认”。当前系统已经具备租户 RBAC、Dify 知识查询、LangGraph Agent、会话记录、AI Run 和回答反馈，但尚未建立 `customer_tickets`、`reply_suggestions` 以及 `/api/v1/customer-service/*` 接口。

需求流程明确规定：客户问题经过分类、知识检索或业务查询、建议回复和质检后，必须由客服人工确认，AI 不得直接代替客服向外部渠道发送。

## 二、需要解决的问题

1. 客服流程由 FastAPI、LangGraph 还是 Dify 作为主要编排器；
2. 企业用户、客服人员、管理员和决策者分别可以创建、查看或处理哪些工单；
3. 首版问题分类和回复生成是否必须再次调用大模型；
4. 如何复用现有 Dify 知识查询，同时避免客服模块直接依赖 Dify 私有协议；
5. 建议回复可以保存多条历史，还是每个工单只保留一条当前建议；
6. 人工确认如何保证幂等，如何防止确认后再次修改或自动发送；
7. 如何处理跨租户访问、同租户普通用户越权、并发生成与并发确认；
8. 教师业务数据尚未下发时，如何保持接口和数据模型可验收且便于以后替换数据。

## 三、可选方案

### 方案 A：模型生成后直接发送

- 优点：链路短，演示时操作最少。
- 缺点：模型可能出现幻觉或不恰当表述；缺少人工责任边界；不符合需求中的 Human-in-the-loop 原则。

### 方案 B：把完整客服流程全部放入 Dify Workflow

- 优点：可视化配置方便，运营人员容易调整。
- 缺点：会与项目已经确定的 LangGraph 主要 Agent 编排职责重叠；租户权限、工单事务和人工确认状态仍必须回到 FastAPI；容易形成两个业务状态源。

### 方案 C：FastAPI 管理业务状态，LangGraph 编排分类、知识检索、回复组合和质检

- 内容：FastAPI 负责鉴权、工单和建议持久化；LangGraph 负责按固定节点执行分类、调用统一知识端口、生成建议和质检；Dify 仍是正式知识源；客服通过独立确认接口完成最终编辑和确认。
- 优点：职责清晰；复用已有租户隔离和 Dify 适配器；图节点可以独立测试；不会让 Dify 与 LangGraph 循环调用。
- 缺点：需要新增客服领域、图状态、仓储事务和较多边界测试。

### 方案 D：每次生成都新增一条建议历史

- 优点：能够比较不同生成版本，审计信息更完整。
- 缺点：当前没有“选择哪一条确认”的前端交互；并发确认需要额外唯一约束；教学首版复杂度较高。

### 方案 E：每个工单维护一条当前建议，确认前允许幂等更新

- 优点：符合当前 API 清单和单一确认控件；并发生成可以通过 `ticket_id` 唯一约束收敛；确认后状态稳定。
- 缺点：V0.1 不保留每次重新生成的历史内容。

## 四、最终决策

采用方案 C 与方案 E，并固定以下规则：

1. 新建独立 `app/customer_service/` 领域，包含模型、Schema、端口、分类策略、LangGraph 工作流、仓储、服务、依赖和路由；
2. 工单状态使用 `open -> in_progress -> resolved`，V0.1 预留 `closed`，但不提供关闭接口；
3. 稳定分类值使用 `refund_after_sales`、`account_security`、`product_service`、`knowledge_document`、`technical_support` 和 `other`；优先级使用 `low|normal|high|urgent`；
4. 任意已登录用户可以为自己创建工单；普通用户和决策者只能查看自己的工单，`customer_service` 与 `admin` 可以查看和处理当前租户全部工单；列表项对所有角色统一使用最小公开结构，客服需要请求者、处理人、分类推理和建议内容时再查询内部详情；公开视图不得包含租户 ID、员工 ID、AI 草稿、质检说明或工作流元数据，只有人工确认后的最终回复可以返回给请求者；
5. 分类、生成建议和人工确认接口只允许 `customer_service` 与 `admin`；跨租户资源、普通用户访问他人工单和不存在资源统一使用 404，角色不具备处理权限时使用 403；
6. LangGraph 顺序固定为 `classify -> retrieve_knowledge -> compose_reply -> quality_check`，不在图内执行发送；
7. 分类节点首版采用稳定、可解释的关键词规则并输出置信度、优先级和理由；通过端口保留以后替换模型分类器的能力；
8. 知识节点只依赖客服知识端口，由适配器调用现有 `KnowledgeService`，不得直接处理 Dify Dataset ID、Document ID、Key 或原始响应；
9. 首版回复组合使用知识查询结果和受控模板，不新增第二次大模型调用，确保答案可追溯、成本可控和真实冒烟稳定；以后可以替换 Reply Composer，而不修改 API 和工单模型；
10. 质检节点只标记 `passed|needs_review` 和公开原因，任何结果都必须人工确认；空召回、拒答或依据不足时必须标记 `needs_review`，不得伪造确定答案；
11. 每个工单只有一条当前 `reply_suggestion`，数据库对 `ticket_id` 建立唯一约束；确认前重复生成原子更新当前建议，确认后禁止重新生成或改写；
12. 确认接口允许客服使用建议原文或提交编辑后的最终回复；生成和确认统一按 `CustomerTicket -> ReplySuggestion` 顺序获取行锁，确认成功后在同一事务中写入确认人和时间，并将工单置为 `resolved`；
13. 对已经确认的建议重复提交相同最终回复时幂等返回原结果，提交不同内容时返回 409，避免“已发送内容”被静默覆盖；
14. V0.1 的“发送”只表示平台内完成确认和留痕，不连接短信、邮件、微信或第三方客服渠道；真实外部发送另立决策并增加渠道回执和失败补偿；
15. `reply_suggestions` 保存 `tenant_id` 作为数据库完整性字段，并通过 `(ticket_id, tenant_id)` 复合外键绑定所属工单；工单请求者、处理人以及建议生成者、确认者都通过 `(user_id, tenant_id)` 复合外键绑定同租户用户，避免脚本、迁移或仓储缺陷写入跨租户关系；
16. 用户删除策略采用数据库 `RESTRICT`：存在客服工单或建议审计记录时不得物理删除关联用户；项目继续使用禁用状态表达离职或停用，已确认建议必须同时具有 `final_reply`、`confirmed_at` 和 `confirmed_by_user_id`；
17. 工单主题、问题正文、建议和最终回复不得写入普通访问日志和决策日志，测试只使用模拟内容。

### 2026-07-17 独立审查修订

初版实现通过功能测试后，独立代码审查发现三个阻塞风险：请求者可在人工确认前读取内部草稿、数据库未强制用户与租户一致、建议重新生成与确认存在相反锁顺序。为此，本决策在迁移正式定稿前补充公开/内部响应分离、租户感知复合外键、统一锁顺序和不可为空的确认人审计约束。该修订不改变六个 API 的业务职责，但会收紧响应字段和数据库不变量。

## 五、决策理由

FastAPI 擅长处理租户权限、状态持久化和事务，LangGraph 适合表达可观察的多步骤流程，Dify 已经是正式知识来源。将三者按职责组合，可以避免把数据库业务状态放进 Dify，也避免在客服路由中直接堆积分类、检索和回复逻辑。

教学首版优先验证完整闭环而不是追求复杂生成。规则分类和知识抽取式模板能够稳定复现、解释结果和执行真实冒烟，同时端口和图节点保留以后接入 DeepSeek 或通义千问分类/生成的替换点。人工确认是最终安全边界，任何质量状态都不能绕过。

## 六、影响与风险

- 正面影响：完成客服核心交付；形成可按租户、分类和状态统计的工单数据；建议回复具备知识依据和人工责任边界。
- 潜在风险：规则分类对表达方式敏感；首版只维护当前建议，不保留生成历史；“确认”尚未连接真实外部发送渠道。
- 控制措施：稳定枚举和分类测试集；知识无答案时显式降级；建议唯一约束；公开/内部 Schema 分离；租户感知复合外键；统一 `Ticket -> Suggestion` 行锁顺序；严格 RBAC 与 404 隐藏；真实双租户冒烟。
- 回退方案：客服路由可以独立移除，不影响现有 AI 对话、Agent、知识库和反馈；迁移支持 downgrade 删除客服表。

## 九、可用于实习日记的素材

客服辅助不是“让模型自动回复”，而是让 AI 提供可追溯建议，并把最终决定保留给客服人员。通过 LangGraph 把分类、检索、回复和质检拆成节点，可以单独观察每一步；通过 FastAPI 和数据库保存人工确认状态，可以明确区分 AI 建议与正式回复。
