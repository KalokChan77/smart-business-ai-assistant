# DEC-20260717-012：采用 Argon2、双 JWT 与 Redis 令牌吊销

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：登录、刷新、退出和当前用户认证
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

技术方案要求后端实施 JWT/OAuth2、RBAC 和接口权限校验。项目既需要普通访问令牌，也需要刷新令牌和退出后的失效记录；同时数据库只能保存密码哈希，不能保存明文密码。

## 二、需要解决的问题

需要确定密码算法、JWT 声明、访问与刷新策略以及退出后的失效方式。如果只签发一个长期 JWT，令牌泄漏后的风险窗口过长；如果完全依赖数据库会话，又不能体现本项目要求的 JWT 教学重点。

## 三、可选方案

### 方案 A：单个长期 JWT，不保存吊销状态

- 优点：实现最简单，不需要 Redis。
- 缺点：无法可靠退出，令牌泄漏后只能等待过期。

### 方案 B：随机会话 Token，全量保存到数据库

- 优点：服务端可以立即控制所有会话。
- 缺点：偏离 JWT 教学要求，每次请求都需要查询会话表。

### 方案 C：短期访问 JWT + 轮换刷新 JWT + Redis 吊销

- 优点：访问令牌有效期短；刷新令牌每次使用后轮换；退出时可以按 `jti` 立即吊销；Redis 记录会随令牌过期自动清理。
- 缺点：认证请求需要依赖 Redis，令牌类型和异常分支更多。

## 四、最终决策

- 密码使用 `pwdlib` 推荐配置进行 Argon2 哈希和验证；
- 访问令牌和刷新令牌使用不同的 `type` 声明和独立 `jti`；
- JWT 必须包含 `sub`、`tenant_id`、`type`、`jti`、`iat`、`exp`、`iss` 和 `aud`；
- 访问令牌默认有效 30 分钟，刷新令牌默认有效 7 天，均可通过服务端配置调整；
- 刷新操作先验证并吊销旧刷新令牌，再签发新令牌对；
- 退出时吊销当前访问令牌和提交的刷新令牌；
- Redis 只保存吊销标记，TTL 等于令牌剩余有效期，不保存完整 JWT；
- 未找到用户时仍执行一次密码哈希验证，降低明显的用户枚举计时差异；
- 日志和错误响应不得记录密码或完整 JWT。

## 五、决策理由

双令牌兼顾教学要求和基本安全性。访问令牌短期有效可以缩小泄漏影响，刷新轮换可以减少长期令牌重复使用。Redis 适合保存短期吊销状态，并能通过 TTL 自动清理，不会把临时会话状态混入长期业务数据。

## 六、影响与风险

- 正面影响：支持登录、刷新、退出和即时失效，令牌职责清晰。
- 潜在风险：Redis 不可用时无法判断令牌是否已吊销。
- 回退方案：认证依赖不可用时返回服务不可用，不采用“Redis 故障就默认放行”的不安全降级。

## 七、验证证据

- 密码测试覆盖正确密码、错误密码和哈希不等于明文；JWT 测试覆盖访问/刷新类型隔离、过期和篡改令牌。
- 登录、刷新、当前用户和退出 API 均通过自动化测试与真实 Uvicorn 验收。
- 刷新令牌通过 Redis `SET NX` 原子消费；同一刷新令牌第一次返回 HTTP 200，重复使用返回 HTTP 401 `invalid_token`。
- 真实 Redis 验证吊销值仅为固定标记 `1`，TTL 大于 0，未保存完整 JWT。
- 退出后访问令牌立即返回 HTTP 401，旧访问令牌和轮换后的访问令牌均完成吊销验证。
- 登录和刷新响应设置 `Cache-Control: no-store` 与 `Pragma: no-cache`。
- 参数校验错误、访问日志和服务端输出均未出现测试密码或完整 JWT。

## 九、可用于实习日记的素材

我学习到 JWT 并不天然支持退出。要实现可控退出，需要给令牌分配唯一 `jti`，并在 Redis 中保存到期前的吊销状态。访问令牌与刷新令牌也必须区分用途，不能让刷新令牌直接访问业务接口。
