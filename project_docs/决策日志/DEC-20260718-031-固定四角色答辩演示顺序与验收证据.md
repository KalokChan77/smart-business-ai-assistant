# DEC-20260718-031：固定四角色答辩演示顺序与验收证据

- 日期：2026-07-18
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：技术方案 P5 答辩演示、模拟验收问题校准
- 关联实习日记：`project_docs/实习日记/2026-07-18-实习日记.md`
- 关联决策：`DEC-20260718-029`、`DEC-20260718-030`

## 一、背景

技术方案 P5 要求完成 Docker Compose、测试、README 和答辩演示。现有模拟流程仍包含尚未实现的订单查询工具、自动备用模型和外部客服发送接口，若直接用于答辩会造成“文档能力大于实际能力”的问题。四角色浏览器冒烟已经证明真实主线可用，因此需要把答辩材料严格绑定当前代码和接口证据。

## 二、需要解决的问题

1. 12 分钟内如何覆盖最有代表性的四角色业务闭环；
2. 哪些提示词能够稳定触发普通 SSE、LangGraph 工具和 Dify 知识问答；
3. 如何解释 AI 建议与人工确认的责任边界；
4. 现场故障时如何切换流程而不暴露敏感信息；
5. 如何清理旧验收材料中未实现的能力描述。

## 三、可选方案

### 方案 A：按功能菜单自由演示

- 优点：现场灵活；
- 缺点：容易超时、漏掉关键闭环，统计数据也可能还未生成。

### 方案 B：固定“管理员 → 用户 → 客服 → 用户 → 决策者”顺序

- 优点：先说明管理与数据基础，再产生业务数据，随后完成客服闭环，最后查看统计；前后关系自然；
- 缺点：需要多次退出登录。

### 方案 C：只播放录屏

- 优点：稳定；
- 缺点：不能证明现场系统可用，只适合作为故障备份。

## 四、最终决策

采用方案 B，并固定：

1. 总时长约 12 分钟；
2. 演示前执行一键启动和状态检查，health/readiness/前端必须 HTTP 200；
3. 普通对话使用一句话平台说明，验证 SSE 与历史持久化；
4. Agent 使用计算题 `(1280+720)*0.85`，验证白名单工具与 `tool_start/tool_end`；
5. 知识问答使用退款期限问题，验证 Dify `answered` 与引用；
6. 客服流程必须走完“用户建单 → AI 分类 → 知识增强建议 → 人工确认 → 用户查看最终回复”；
7. 决策者最后展示统计，确保之前产生的数据已经进入聚合；
8. 删除或改写未实现的订单查询、自动备用模型和外部发送描述；
9. 录屏仅作为现场故障备份，不替代真实演示。

## 五、决策理由

1. **真实性**：每个展示步骤都能在当前代码、API 契约或浏览器证据中找到对应实现；
2. **叙事完整**：从配置管理、用户提问到客服确认和经营统计形成完整业务故事；
3. **风险可控**：提示词、顺序和故障替代路径固定，减少现场随机性；
4. **安全**：演示流程明确禁止投屏 `.env`、Key、Token、密码和 Dataset ID。

## 六、影响与风险

- 正面影响：技术方案 P5 形成可直接执行的答辩材料；模拟验收问题与真实工具保持一致；
- 潜在风险：真实模型响应受网络和额度影响，Dify 容器重建后可能出现 Nginx 旧上游地址；
- 回退方案：保留历史会话、已有知识回答和录屏作为展示证据，但现场明确说明当前上游状态。

## 七、验证证据
