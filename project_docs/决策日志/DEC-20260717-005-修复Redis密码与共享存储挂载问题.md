# DEC-20260717-005：修复 Redis 密码与共享存储挂载问题

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Dify 文档上传与异步索引故障排查
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

Dify 控制台和基础 API 已经运行，但第一次上传知识文档时出现 HTTP `500`。修复 Redis 后，文档仍然出现 `File not found`，说明异步索引 Worker 无法读取 API 保存的文件。

## 二、需要解决的问题

需要分别定位消息队列连接失败和文件共享失败的根因，确保后续原生文件上传可以自动完成索引，而不是依赖人工复制文件。

## 三、可选方案

### 方案 A：绕过异步索引，仅手动写入文件

- 优点：能够快速完成一次性演示。
- 缺点：不能支持后续正常上传，不是真正解决问题。

### 方案 B：修正 Redis 与 Compose 存储配置

- 优点：恢复 Dify 标准上传和 Worker 索引流程，后续可以重复使用。
- 缺点：需要分析容器环境变量、挂载目录、数据库记录和 Worker 日志。

## 四、最终决策

采用根因修复方案：

1. 将 `CELERY_BROKER_URL` 中的 Redis 密码与实际 `REDIS_PASSWORD` 保持一致；
2. 使用绝对共享存储路径 `/app/api/storage`；
3. 新增 `docker-compose.override.yaml`，为 `api_websocket` 服务挂载与 API、Worker 相同的存储目录；
4. 保持敏感 Docker 环境文件权限为 `0600`。

## 五、决策理由

日志显示 Celery 因密码错误无法连接 Redis；进一步隔离测试发现上传文件实际写入 `api_websocket` 容器，但该容器没有共享存储挂载。只有同时修复消息队列和存储挂载，才能恢复完整的异步索引链路。

## 六、影响与风险

- 正面影响：原生文件上传、共享存储和异步索引全部恢复正常。
- 潜在风险：未来升级 Dify 时，官方 Compose 文件结构可能变化，需要检查 override 是否仍然适用。
- 回退方案：升级前备份 `.env`、override 和 `volumes`，升级后重新执行上传与索引测试。

## 七、验证证据

- Worker 日志显示成功连接 Redis，并进入 `ready` 状态。
- 原生上传接口返回 HTTP `200`。
- 上传文件能够在共享存储目录中找到。
- 测试文件状态从 `indexing` 变为 `completed`，耗时约 2 秒。
- 清理测试文件后，正式知识库保持 5 份文档、5 份完成、0 份错误。
