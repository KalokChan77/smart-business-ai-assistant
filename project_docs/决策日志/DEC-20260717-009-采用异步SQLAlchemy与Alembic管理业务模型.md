# DEC-20260717-009：采用异步 SQLAlchemy 与 Alembic 管理业务模型

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：数据库访问层、用户角色模型和迁移
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

FastAPI 后续需要处理流式 AI 请求和多个外部 HTTP 调用。数据库访问如果长期使用同步阻塞方式，容易占用事件循环。与此同时，用户、角色和后续业务表需要通过版本化迁移创建，而不能依赖应用启动时自动建表。

## 二、需要解决的问题

需要确定数据库会话方式、模型组织方式和结构变更流程，使领域模型保持高内聚，同时让迁移可追踪、可升级和可回退。

## 三、可选方案

### 方案 A：同步 SQLAlchemy + 启动时 `create_all`

- 优点：实现简单。
- 缺点：阻塞异步接口；`create_all` 不能可靠管理字段修改、索引变化和回退。

### 方案 B：异步 SQLAlchemy + Alembic

- 优点：与 FastAPI 异步模型一致；迁移文件能够记录数据库结构演进；便于测试和部署验收。
- 缺点：配置和测试比同步方式稍复杂。

## 四、最终决策

- 使用 SQLAlchemy 2.x 异步引擎和 `async_sessionmaker`；
- 使用 psycopg 3 作为 PostgreSQL 驱动，并显式声明异步引擎所需的 `greenlet` 依赖；
- 使用 Alembic 管理数据库迁移，连接地址只从 `Settings` 和环境变量读取，不在 `alembic.ini` 保存密码；
- 模型按业务领域存放，例如用户与角色模型放在 `app/users/`，公共 Base 与会话工厂放在 `app/db/`；
- 健康检查通过抽象探针访问引擎，不让 HTTP 路由直接依赖 SQLAlchemy；
- 第一版创建 `users`、`roles` 和 `user_roles`，并预留 `tenant_id`。

## 五、决策理由

异步引擎适合 FastAPI 的请求模型，Alembic 能够提供真实的数据库版本证据。按领域组织模型可以让用户相关代码集中维护；公共数据库模块只承担连接、Base 和会话生命周期，避免形成包含全部业务模型的“万能数据库层”。

## 六、影响与风险

- 正面影响：数据库结构可演进，接口不会被同步连接长期阻塞。
- 潜在风险：异步会话使用不当可能出现事务未提交或连接未关闭。
- 回退方案：统一通过依赖或应用服务管理会话，测试事务边界；不在路由中直接创建引擎。

## 七、验证证据

- 首次执行异步迁移时发现缺少 `greenlet`，随后将其加入 `pyproject.toml` 后重新安装，避免只修复本机而遗漏项目依赖声明。
- Alembic 初始迁移成功升级到版本 `5a4168ddc1f8`，并完成一次 `downgrade base` 后再 `upgrade head` 的往返验证。
- `alembic check` 返回“No new upgrade operations detected”，说明模型元数据与数据库结构一致。
- PostgreSQL 中存在 `users`、`roles`、`user_roles` 和 `alembic_version`。
- 数据库探针执行 `SELECT 1` 成功，Redis 探针执行 `PING` 成功。
- FastAPI 真实启动后 readiness 返回 HTTP 200，OpenAPI 文档正常生成。
- Python 编译检查通过，自动化测试共 4 项通过且无警告。
