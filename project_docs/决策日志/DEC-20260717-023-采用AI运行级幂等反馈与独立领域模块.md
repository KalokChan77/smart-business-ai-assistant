# DEC-20260717-023：采用 AI 运行级幂等反馈与独立领域模块

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：AI 回答赞踩、文字反馈、消息与 Run 关联、满意度数据基础
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

需求基线要求优先完成“登录 -> 对话 -> AI 路由 -> 工具/知识检索 -> 流式回复 -> 记录落库 -> 用户反馈”的完整闭环。当前系统已经实现认证、会话、消息、普通模型与 LangGraph Agent 流式运行、AI Run 查询和知识库能力，但 `POST /api/v1/ai/runs/{run_id}/feedback` 仍停留在 API 清单中，数据库也没有反馈模型。

模拟验收问题明确要求：用户对回答点击“不满意”后，系统必须保存反馈并同时关联助手消息和 AI Run。后续满意度统计也需要稳定、可按租户聚合的反馈数据源。

## 二、需要解决的问题

1. 反馈数据放入 AI 模块、会话模块，还是建立独立领域模块；
2. 前端应提交 Run ID、Message ID，还是只提交 Run ID 并由服务端确定回答消息；
3. 用户重复点击赞或踩、从赞改为踩时，是新增多条记录还是更新当前反馈；
4. 哪些 AI Run 可以评价，如何处理 running、failed、cancelled 或没有回答消息的 Run；
5. 如何保证其他租户、同租户其他用户和管理员不能代替 Run 所有者提交反馈；
6. 如何让反馈数据能够支持后续满意度统计，同时不保存不必要的冗余字段和敏感内容。

## 三、可选方案

### 方案 A：把反馈字段直接增加到 `ai_runs`

- 优点：查询一次即可得到评价，表结构最少。
- 缺点：把运行生命周期和用户评价混在同一模型；评论、评价时间和后续反馈扩展会不断膨胀 `ai_runs`；无法清晰表达反馈是独立业务实体。

### 方案 B：每次点击都新增一条不可变反馈事件

- 优点：能够保留完整修改历史，适合精细审计。
- 缺点：前端一次误触会产生多条有效记录；满意度统计必须额外计算“每个 Run 的最新一条”；当前教学阶段复杂度过高。

### 方案 C：建立独立反馈领域，每个 Run 保存一条可幂等更新的当前反馈

- 内容：新建 `app/feedback/`；反馈表保存 Run、回答消息、评价、可选评论和时间；租户与用户归属统一通过 Run 推导；数据库对 `run_id` 建立唯一约束；重复 POST 使用 upsert 更新当前反馈。
- 优点：反馈职责独立；同一 Run 只有一个当前评价，适合前端切换赞踩和后续满意度统计；可以单独扩展审核、标签或反馈历史。
- 缺点：当前不保留每一次赞踩切换历史；如果以后需要完整审计，需要新增反馈事件表。

### 方案 D：允许前端同时提交 `run_id` 和 `message_id`

- 优点：前端可以明确指出被评价的消息。
- 缺点：客户端可能提交不属于该 Run、其他会话或其他用户的 Message ID，服务端必须增加复杂的一致性校验，也扩大越权输入面。

### 方案 E：只接收 `run_id`，由服务端绑定 `response_message_id`

- 优点：回答消息来源唯一且可信；前端不能伪造关联；失败或未完成 Run 可以根据服务端状态稳定拒绝。
- 缺点：当前只支持评价 AI Run 的最终助手回答，不支持任意历史消息评价。

### 方案 F：只在应用仓储中校验 Run 与 Message 关联

- 优点：数据库结构简单，所有业务规则都集中在 Python 代码中。
- 缺点：未来脚本、迁移或新增仓储路径如果绕过当前服务，仍可能插入“Run 属于一个回答、反馈却指向另一个 Message”的数据；已有反馈后若修改 Run 的最终回答，也可能产生陈旧关联。

### 方案 G：应用校验之外，再增加 Run/Message 复合数据库不变量

- 内容：在 `ai_runs(id, response_message_id)` 上建立复合唯一键，反馈使用 `(run_id, message_id)` 复合外键引用它；已有反馈时禁止静默替换该 Run 的最终回答，重新生成回答必须创建新的 AI Run。
- 优点：直接写表、未来脚本和新仓储也无法绕过核心关联；反馈评价对象保持稳定，满意度统计不会随着事后修改而改变口径。
- 缺点：增加一个复合约束；如果业务未来确实需要修改已评价回答，必须先明确迁移反馈或采用新 Run，而不能直接覆盖字段。

## 四、最终决策

采用方案 C、方案 E 与方案 G，并固定以下规则：

1. 新建独立 `app/feedback/` 领域，包含模型、Schema、仓储、服务、依赖和路由；AI 流式生成模块不直接承担反馈持久化；
2. 对外接口固定为 `POST /api/v1/ai/runs/{run_id}/feedback`，任何已登录角色都可以评价自己创建的 Run；管理员不具备跨用户代提交权限；
3. 请求体只允许 `rating` 和可选 `comment`，拒绝 `message_id`、`user_id`、`tenant_id` 及其他未声明字段；
4. `rating` 使用稳定业务值 `positive|negative`；`comment` 去除首尾空白后最多 1000 字符，空字符串保存为 `null`；
5. 只有 `status=succeeded` 且存在 `response_message_id` 的 Run 可以评价；running、failed、cancelled 或状态不完整时返回 HTTP 409 `ai_run_not_feedbackable`；
6. Run 必须同时匹配当前 `tenant_id` 和 `user_id`；其他租户、同租户其他用户或不存在的 Run 统一返回 HTTP 404 `ai_run_not_found`；
7. 服务端从 Run 读取 `response_message_id`，前端不能指定回答消息；反馈同时关联 Run 和该回答消息；
8. 每个 Run 只有一条当前反馈，数据库对 `run_id` 建立唯一约束；仓储在同一事务中锁定并校验 Run、回答消息和所属会话后再执行 upsert，防止校验与写入之间的状态变化；
9. 反馈表只保留 `run_id`、`message_id`、评价、评论和时间，不重复保存 `tenant_id`、`user_id`、conversation、provider、model 或回答正文；所有权与租户统计统一连接 AI Run 推导，消除冗余归属字段漂移；
10. 可反馈目标必须满足：Run 属于当前主体、状态为 succeeded、回答消息属于同一会话且角色为 assistant、会话仍属于当前主体且未软删除；
11. 数据库使用复合唯一键和复合外键强制 `ai_feedback(run_id, message_id)` 与 `ai_runs(id, response_message_id)` 一致；已有反馈后不能直接替换该 Run 的最终回答，重新生成或修正回答必须创建新的 Run；
12. PostgreSQL 使用 `INSERT ... ON CONFLICT DO UPDATE ... RETURNING` 原子收敛重复提交，冲突更新显式同步当前 `message_id`，并使用 `populate_existing=True` 刷新 SQLAlchemy Identity Map，避免同一 Session 中更新后仍读取旧评价；
13. 建立创建时间与评价组合索引；后续满意度统计通过 `feedback.run_id -> ai_runs` 连接应用租户范围；
14. 响应不返回租户 ID、用户 ID、回答正文、模型凭据或内部错误；评论不得进入访问日志；
15. 本阶段只维护当前反馈，不实现反馈修改历史、举报审核和管理员导出，这些能力另立决策。

## 五、决策理由

反馈与 AI Run 生命周期有关，但它属于用户评价业务，不属于模型调用和 SSE 状态机。建立独立领域可以让 AI 模块继续专注运行创建、流式输出和结果持久化，反馈模块只依赖 Run 的稳定所有权和回答消息字段，符合高内聚、低耦合原则。

由服务端绑定回答消息可以避免客户端构造跨会话 Message ID。每 Run 一条原子 upsert 既符合赞踩控件可以切换的交互习惯，也能让满意度统计直接按当前评价聚合，避免教学项目过早引入事件溯源复杂度。独立审查进一步证明，只依赖仓储校验仍不足以保护长期数据一致性，因此把 Run 与最终回答的配对下沉为数据库复合外键；已有评价的回答不再允许被原地替换，避免历史反馈的评价对象发生变化。

## 六、影响与风险

- 正面影响：完成核心对话闭环；反馈可靠关联 Run 和回答消息；重复点击不会产生重复有效记录；为满意度统计提供租户级数据源。
- 潜在风险：当前只保存最后一次反馈，不保留切换历史；知识查询接口尚未创建 AI Run，因此首版反馈覆盖普通 Chat 和 Agent Run，不直接覆盖独立知识查询响应。
- 控制措施：每 Run 唯一反馈、Run/Message 复合外键；同一事务中的 Run 行锁、消息角色与会话归属检查和原子 upsert；严格 Pydantic 请求；API、服务、并发和真实 PostgreSQL 集成测试；后续若知识查询纳入统一 Run 生命周期，可直接复用同一反馈接口。
- 回退方案：如反馈路由出现问题，可暂时从总路由移除，不影响现有对话、Agent 和知识库查询；迁移支持 downgrade 删除反馈表。
