# DEC-20260717-001：采用本地 Docker Compose 部署 Dify

## 一、背景

项目需要整合 Vue3、FastAPI、多 AI 服务编排、Dify、DeepSeek 和阿里云百炼。当前目标是完成本科集中实训期间可演示、可测试的原型，不要求大规模并发运行。开发机已经安装 Docker 与 Docker Compose，并具备 16GB 内存，适合运行 Dify 自托管版本。

## 二、需要解决的问题

需要选择 Dify 的运行位置与部署方式，使小组能够独立配置应用、工作流、知识库和 Service API Key，同时保证部署步骤可复现。

## 三、可选方案

### 方案 A：直接使用云端 Dify 服务

- 优点：部署工作少，启动速度快。
- 缺点：受云端额度和账号权限限制，不符合小组自行部署要求，难以展示完整部署能力。

### 方案 B：本地 Docker Compose 自托管

- 优点：数据和配置可控，适合教学演示，可完整学习数据库、Redis、Worker、插件和向量库的协作方式。
- 缺点：镜像较大，对本地磁盘、网络和内存有一定要求。

### 方案 C：直接在 Oracle Cloud 服务器部署

- 优点：能够绑定域名，适合远程展示，具有加分价值。
- 缺点：初期排障成本较高，会引入防火墙、HTTPS、域名解析和公网安全问题。

## 四、最终决策

第一阶段使用官方 Dify `1.15.0` Docker Compose 在本地部署；Oracle Cloud 与域名部署作为后续加分项，不阻塞本地开发和验收。

## 五、决策理由

本地部署最符合“测试成功即可”的实训目标，能够先验证核心功能，再逐步扩展到服务器。Docker Compose 也便于统一 PostgreSQL、Redis、Weaviate、Dify API、Web、Worker、Sandbox 和插件守护进程的版本与网络关系。

## 六、影响与风险

- 正面影响：开发环境独立可控，API 和知识库数据均保存在本机。
- 潜在风险：Docker Hub 网络不稳定、镜像体积较大、容器资源占用较高。
- 回退方案：使用镜像加速器拉取基础镜像；服务器部署延后到本地验收完成后。

## 七、验证证据

- Dify 控制台可通过 `http://localhost:18080` 访问。
- Dify API、PostgreSQL、Redis、Sandbox 等核心容器均正常运行。
- Dify API 容器健康检查结果为 `healthy`。
