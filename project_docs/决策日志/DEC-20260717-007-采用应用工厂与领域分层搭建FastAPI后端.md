# DEC-20260717-007：采用应用工厂与领域分层搭建 FastAPI 后端

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：技术方案 P0 第一阶段 FastAPI 基础
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

技术方案要求所有前端请求统一进入 FastAPI，并明确规定 API 路由只负责参数校验和调用应用服务，不应在路由函数中堆积数据库 SQL、提示词或 Agent 图。项目目前只有技术文档和 Dify 环境，尚未建立业务后端代码。

## 二、需要解决的问题

需要确定首版 FastAPI 工程结构，使后续认证、数据库、会话、AI Gateway、Dify、LangGraph、知识库和客服模块能够按顺序扩展，同时避免所有逻辑集中在单个 `main.py` 中。

## 三、可选方案

### 方案 A：单文件 FastAPI 应用

- 优点：代码少，首个接口启动快。
- 缺点：后续路由、配置、数据库和 AI 逻辑容易混杂，不符合高内聚低耦合要求。

### 方案 B：按技术层建立 controllers/services/models 大目录

- 优点：传统分层清晰。
- 缺点：同一业务功能被拆散到多个全局目录，项目扩大后查找和修改成本较高。

### 方案 C：应用工厂 + 领域模块 + 薄 API 路由

- 优点：各领域内部保持高内聚，通过明确接口与依赖注入降低耦合；应用工厂便于测试时替换配置和外部依赖。
- 缺点：初期文件数量比单文件方案多。

## 四、最终决策

采用应用工厂与领域分层：

```text
backend/
  app/
    api/v1/       # API 聚合与版本前缀
    core/         # 配置等横切能力
    health/       # 健康检查领域
    main.py       # 应用工厂和 ASGI 入口
  tests/          # 自动化测试
```

健康检查领域独立维护响应模型、依赖探针和服务逻辑。路由只负责调用服务并根据结果设置 HTTP 状态。数据库和 Redis 在第二阶段通过探针接口接入，不修改健康检查路由契约。

## 五、决策理由

应用工厂允许测试代码创建独立 FastAPI 实例并注入替代依赖。领域模块让健康检查、认证、会话、AI 和知识库代码各自聚合。依赖探针使用 Protocol 抽象，后续接入 PostgreSQL 和 Redis 时只需要新增实现，不需要让健康检查模块直接依赖具体数据库驱动。

## 六、影响与风险

- 正面影响：代码结构可扩展、便于单元测试和接口测试，符合高内聚低耦合原则。
- 潜在风险：如果为简单功能创建过多抽象，会增加理解成本。
- 回退方案：只为具有替换需求的外部依赖定义接口，普通数据转换保持直接实现，避免过度设计。

## 七、验证证据

- `/api/v1/health` 应返回 HTTP 200。
- `/api/v1/health/ready` 在数据库和 Redis 尚未接入时应诚实返回 HTTP 503，并指出待完成依赖。
- 测试可注入通过的探针，使 readiness 返回 HTTP 200，证明接口与具体实现解耦。
- `uvicorn app.main:app --reload` 能正常启动。
