# DEC-20260717-026：采用租户实时聚合实现基础统计

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：咨询趋势、问题分类、满意度、人工接管率、AI Run 运行统计与高频问题
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

需求基线把“AI 运行日志、反馈和基础统计”列入核心交付范围，并要求管理员和决策者能够查看咨询量趋势、问题分类分布、满意度、人工接管率、高频问题以及模型成功率、耗时和错误统计。当前系统已经具备租户用户、客服工单、AI Run、反馈、会话和人工确认数据，但尚未提供 `/api/v1/analytics/*` 接口。

项目定位是本地单机教学演示，以测试成功和可解释指标为主，不要求建设大数据仓库、实时流处理或复杂 BI 平台。因此首版应优先复用业务 PostgreSQL 中已有事实数据，并明确统计口径、权限、时间范围和空数据行为。

## 二、需要解决的问题

1. 基础统计采用实时 SQL 聚合、定时汇总表还是外部 BI；
2. “咨询量”“满意度”“人工接管率”和“AI 成功率”的分母分别如何定义；
3. 管理员、决策者、客服和普通用户分别能查看哪些统计；
4. 如何保证每个统计查询只读取当前租户数据；
5. 时间参数使用本地时区还是统一时区，默认范围和最大范围是多少；
6. 高频问题是否返回原始工单内容，如何减少个人信息暴露；
7. 错误统计是否返回模型供应商原始错误正文；
8. 空数据、正在运行的 AI Run、未分类工单和未提交反馈如何计入指标；
9. 是否需要为首版新增统计表、缓存或异步任务；
10. 摘要卡片是否再次调用大模型生成。

## 三、可选方案

### 方案 A：每次请求直接从业务表实时聚合

- 优点：不新增数据同步链路；统计结果与当前业务数据一致；便于通过真实 PostgreSQL 集成测试验证租户隔离和口径。
- 缺点：数据量很大时查询成本会增加；多个页面同时刷新会重复聚合。

### 方案 B：新增日汇总表并由定时任务更新

- 优点：查询快，适合长时间范围和大数据量。
- 缺点：需要处理补算、延迟、失败恢复和口径版本；当前教学规模下复杂度高于收益。

### 方案 C：接入外部 BI 或分析数据库

- 优点：图表和多维分析能力完整。
- 缺点：增加部署组件、数据同步与权限面；不属于核心验收要求。

### 方案 D：调用大模型直接阅读数据并生成统计

- 优点：可以输出自然语言总结。
- 缺点：数值不可保证准确；增加额度、延迟和数据泄露风险；不适合作为指标权威来源。

### 方案 E：SQL 计算权威指标，规则生成摘要卡片

- 优点：数值可复现，摘要无需模型额度；能够满足教学演示中的洞察展示。
- 缺点：摘要表达相对固定，不能替代复杂分析报告。

## 四、最终决策

采用方案 A 与方案 E，并固定以下规则：

1. 新增独立 `app/analytics/` 领域，包含 Schema、统计端口、SQLAlchemy 仓储、服务、依赖和路由，不新增业务表和 Alembic 迁移；
2. 实现五个只读接口：`GET /analytics/overview`、`/consultations`、`/categories`、`/satisfaction`、`/ai-runs`；
3. 所有接口只允许当前租户的 `admin` 或 `decision_maker` 角色访问，普通用户和客服人员返回 403；
4. 所有查询必须显式包含 `tenant_id` 条件。反馈统计通过 `ai_feedback -> ai_runs` 关联推导租户，不在反馈表重复保存租户字段；
5. 公共查询参数为可选 `start_date` 与 `end_date`，使用 UTC 自然日、首尾日期均包含；默认最近 30 个 UTC 自然日，最大 366 天；开始日期晚于结束日期或范围超限返回 HTTP 422；
6. “咨询量”定义为时间范围内创建的 `customer_tickets` 数量；趋势按工单创建日期分桶，并补齐无数据日期为 0；
7. “已解决咨询”定义为当前状态为 `resolved|closed` 的范围内工单；“人工接管”定义为 `assigned_user_id` 非空，接管率为人工接管工单数除以咨询总数；
8. 分类分布包括六个稳定分类和 `unclassified`，百分比以范围内全部咨询为分母，四舍五入到两位小数；
9. 满意度以范围内创建的 `ai_feedback` 为准，满意度为 `positive / (positive + negative)`；没有反馈时返回 0，不返回空值或伪造结论；
10. AI 成功率只使用终态 Run 作为分母，即 `succeeded / (succeeded + failed + cancelled)`；`running` 单独返回但不计入成功率；
11. AI 平均耗时只统计 `completed_at >= started_at` 的终态 Run；Token 平均值只统计非空值；所有比率和平均值均返回非负数并统一保留两位小数；
12. AI 错误统计只返回稳定 `error_code` 和数量，不返回 `error_message`、供应商响应正文、Prompt、回答内容或 Provider Request ID；缺少错误码的失败 Run 归入 `unknown`；
13. 高频问题只按当前租户工单 `subject` 的完全相同文本聚合，至少出现 2 次才返回，最多 5 条，截断为 120 字符；不返回描述、用户、工单 ID 或单次问题；
14. 总览摘要卡片根据 SQL 指标以确定性规则生成，不再次调用 DeepSeek、Dify 或其他大模型；
15. 无数据范围返回 HTTP 200 和完整零值结构，方便前端稳定渲染；数据库查询失败统一转换为脱敏的 HTTP 503 `analytics_unavailable`；
16. V0.1 不统计 TTS 次数与费用，因为语音阶段没有持久化音频调用记录；如以后需要，应新增明确的审计事件，而不是从访问日志推断。

## 五、决策理由

1. **准确性**：指标直接由数据库事实计算，能够通过固定测试数据重复验证；
2. **安全性**：严格租户条件、角色限制和聚合响应避免暴露个人记录及模型错误正文；
3. **低复杂度**：教学规模无需额外汇总表、消息队列和 BI 服务；
4. **高内聚低耦合**：统计 SQL 集中在仓储，口径与比率集中在服务，路由只负责权限和响应；
5. **可演示性**：五个接口可以直接支撑 Vue3 仪表盘的指标卡、折线图、分类图和错误表格；
6. **可演进性**：未来数据量增大时，可以在不修改公开 API 契约的前提下把仓储替换为汇总表或分析数据库。

## 六、影响与风险

- 正面影响：管理员和决策者获得可复现的租户统计；现有工单、反馈和 AI Run 数据形成完整业务闭环；前端无需理解数据库结构。
- 潜在风险：实时聚合在数据量增大后可能变慢；工单当前状态会影响历史区间的“已解决”与“人工接管”队列口径；完全相同主题的高频问题可能漏掉语义相近表达。
- 回退方案：移除 analytics 路由即可停止功能，不涉及数据库结构回退；后续可以新增口径版本或汇总仓储替代当前实现。

## 七、验证证据

- 需求基线已确认统计核心范围和决策者页面指标；
- 当前数据模型已经具备租户工单、分类、处理人、AI Run 状态/耗时/Token、反馈评价和稳定错误码；
- Analytics 定向测试共 25 项并全部通过，覆盖日期校验、管理员与决策者权限、普通用户与客服越权、空数据、七类分类、满意度、人工接管率、终态 AI 成功率、模型分组、耗时、Token 平均值、稳定错误码、高频问题和跨租户隔离；
- 真实 PostgreSQL 集成测试验证了 `start_at` 起始边界包含、`end_at` 结束边界排除、反馈经 AI Run 推导租户、缺失错误码归入 `unknown`，以及其他租户工单、反馈和 Run 不进入结果；
- 真实 FastAPI 双租户冒烟完成三类角色登录、管理员与决策者访问五个接口、普通用户返回 403、额外查询字段返回 422，并核对咨询量、分类、满意度、AI Run 与高频问题；退出后临时工单、Run、会话、用户、角色和 Redis 测试键均清理为 0；
- 后端全量回归共收集 288 项测试并全部通过，`compileall`、`pip check`、Alembic 结构漂移检查均通过，当前迁移为 `2b67c8e7c1ea (head)`；
- 独立代码复核确认所有统计 SQL 均显式包含租户条件，响应不暴露租户 ID、用户描述、消息、反馈评论、`error_message`、Run ID 或供应商请求 ID；初次复核提出的 1 个 MEDIUM 文档闭环问题和 2 个 LOW 测试覆盖问题均已整改，第二轮复核结论为 `APPROVE`，HIGH/MEDIUM/LOW 均为 0；
- 冒烟和测试输出仅记录聚合数量与状态，不记录用户问题正文、反馈评论、模型回答、错误正文、Token、Cookie 或任何 API Key；
- 已知剩余风险为教学规模下实时聚合尚未做大数据量压测、历史区间的已解决/人工接管仍按工单当前状态计算，以及完全相同主题无法合并语义相近问题；这些风险不影响本地功能验收。
