# DEC-20260717-020：采用本地文档台账与 Dify 异步索引任务

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：PDF/DOCX/TXT 上传、Dify Dataset 索引、任务状态、删除与重新索引
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

`POST /api/v1/knowledge/query` 已经通过 FastAPI 服务端统一访问 Dify economy 知识库。下一步需要让管理员通过平台上传教师后续提供的 PDF、DOCX、TXT 文档，并能够查看本地台账、Dify 异步索引进度、失败原因以及执行删除和重新索引。

Dify 1.15.0 本地源码确认：文件上传使用 `POST /v1/datasets/{dataset_id}/document/create-by-file`，表单字段为 `file` 和可选 JSON 字符串 `data`；响应包含 `document` 与 `batch`；索引状态使用 `GET /v1/datasets/{dataset_id}/documents/{batch}/indexing-status`，状态依次可能为 `waiting`、`parsing`、`cleaning`、`splitting`、`indexing`、`completed` 或 `error`；文档资源支持 GET、PATCH 文件更新和 DELETE。

## 二、需要解决的问题

1. FastAPI 是否只把文件转发给 Dify，还是同时保存本地文档与任务台账；
2. 如何在不引入 Celery 的教学单机阶段表示异步索引进度；
3. 如何校验扩展名、MIME、文件签名、压缩包和解析结果，拒绝 EXE、伪造 PDF、损坏 DOCX 和超大文件；
4. 如何支持 Dify 重新索引，而不要求管理员再次选择原文件；
5. 如何保证普通用户不能修改知识库，且前端看不到存储路径、Dify Key、Dataset ID、Dify Document ID 和 batch；
6. Dify 调用成功但本地数据库更新失败、或本地成功但 Dify 失败时如何保留可审计状态。

## 三、可选方案

### 方案 A：FastAPI 只做代理，不保存本地文件和数据库记录

- 优点：代码和数据库最简单。
- 缺点：无法形成 `knowledge_documents`、`knowledge_sync_jobs`；不能保存上传者、校验结果和失败任务；重新索引必须重新上传原文件；平台列表完全依赖 Dify 原始协议。

### 方案 B：本地保存文件和台账，使用 Celery 后台轮询 Dify

- 优点：任务可以自动持续刷新，接近生产系统。
- 缺点：当前项目已经由 Dify 自己承担异步索引，再增加业务 Celery 会扩大部署和排错范围；本地测试成功目标不需要双重任务队列。

### 方案 C：本地保存文件和台账，由状态查询接口按需刷新 Dify

- 内容：上传时完成文件安全校验、原文件本地保存、文档与任务记录创建以及 Dify 异步任务提交；前端查询任务时，FastAPI 使用 batch 刷新 Dify 状态并同步本地数据库。
- 优点：满足数据模型、审计、重新索引和状态机要求；不增加新的基础设施；自动化测试容易使用 Mock Client；真实演示可以轮询直到 completed。
- 缺点：如果前端长期不查询，数据库中的镜像状态不会自动刷新；进程在跨系统提交之间崩溃时仍可能产生需要人工清理的不一致。

## 四、最终决策

采用方案 C，并固定以下规则：

1. 新建 `knowledge_documents` 和 `knowledge_sync_jobs`，业务 PostgreSQL 保存平台台账，Dify 继续保存正式知识索引；不建立第二套向量索引；
2. `knowledge_documents` 保存租户、上传者、原文件名、媒体类型、扩展名、大小、SHA-256、本地 UUID 存储键、当前业务状态、Dify 状态镜像和安全错误码；
3. `knowledge_sync_jobs` 保存 `upload|reindex|delete` 操作及 `pending|processing|completed|failed` 状态、公开进度和脱敏错误；
4. 原文件保存在后端专用忽略目录，文件名使用租户 UUID 与文档 UUID，不使用用户原文件名作为路径；写入采用临时文件后原子替换，并限制文件权限；
5. 首版只允许 `.pdf`、`.docx`、`.txt`，单文件最大 15 MiB；文件名不得包含路径分隔符和控制字符；扩展名、MIME、魔数/ZIP 结构和解析器结果必须一致；
6. PDF 使用 `pypdf` 校验头、加密状态和可解析页数；DOCX 使用 ZIP 安全检查与 `python-docx` 解析；TXT 必须可按 UTF-8/UTF-8-SIG 解码、无 NUL 且包含非空文本；
7. DOCX 压缩包限制成员数量和解压后总大小，不把压缩包解压到文件系统；
8. 上传调用 Dify canonical hyphen 路径，表单 `data` 固定使用 `indexing_technique=economy`、`doc_form=text_model`、`doc_language=Chinese`、`process_rule.mode=automatic`；
9. 上传成功后保存 Dify Document ID 与 batch，但这些内部字段不进入平台公开响应；
10. `GET /api/v1/knowledge/jobs/{job_id}` 对 processing 任务按需调用 Dify indexing-status，并把 `waiting/parsing/cleaning/splitting/indexing/paused` 映射为 processing，把 completed 映射为 completed，把 error/stopped 映射为 failed；
11. 重新索引读取本地原文件，通过 Dify 文档 PATCH 文件更新接口创建新 batch；删除采用可恢复的分阶段顺序：先删除 Dify 文档并持久化清空内部文档标识，再删除本地文件，最后软删除台账；
12. 上传、列表、详情、任务、删除和重新索引全部限制为 `admin` 角色；普通用户只使用知识查询；
13. 本阶段 Dify Dataset 是单机教学部署的共享正式知识源；未登记到业务台账的预置教学文档继续共享，通过 FastAPI 上传并登记的托管文档在回答前按本地 Dify Document ID 台账执行租户过滤，已软删除托管文档不再可见；
14. 不记录文件正文、API Key、Dataset ID、Dify batch、内部存储绝对路径和第三方原始错误正文；
15. 当前不引入业务 Celery；未来需要自动刷新、批量处理或断点恢复时再升级为后台队列；
16. Dify mutation 成功但本地首次保存失败时，文档和任务必须落为可恢复的失败态，保留仅供服务端恢复使用的上游文档标识和 batch；连续保存失败时，对新上传的上游文档尝试补偿删除；
17. pending/processing 任务缺少 batch 且超过 300 秒时，由任务查询接口标记为 `failed` 并解除 active job 阻塞；
18. 删除任务记录 `dify_deleted_at` 和 `file_deleted_at` 两个内部阶段；Dify 已删除但文件删除失败时，再次 DELETE 只继续剩余阶段，不重复使用陈旧上游文档标识。

## 五、决策理由

本地台账与 Dify 正式索引分工明确：PostgreSQL 负责业务身份、权限、审计和任务状态，Dify 负责文档切分、关键词索引和检索。保存原文件可以支持重新索引和问题复现，而 UUID 存储路径与严格解析可以降低路径穿越、扩展名伪造和压缩包风险。

按需刷新状态适合本次“本地测试成功”的教学规模，能完整展示异步状态机，又不会重复部署另一套 Celery。通过端口和适配器抽象，未来仍可把状态刷新迁移到后台任务，不改变 API 和数据库主模型。

## 六、影响与风险

- 正面影响：形成完整上传、台账、任务、索引、重新索引和删除闭环；教师文档到达后可直接替换测试文件；前端不依赖 Dify 原始协议。
- 潜在风险：跨 PostgreSQL、本地文件系统和 Dify 的操作不能使用单一数据库事务；极端进程崩溃仍可能留下待清理资源；预置文档继续共享，应用层过滤也不能替代真正的索引级租户隔离；解析器不能代替专业杀毒引擎。
- 控制措施：每一步保留台账、失败状态和删除阶段；唯一 active job 约束防止同一文档并行变更；托管文档在生成回答前执行租户可见性过滤；连续保存失败时尝试补偿删除；所有操作支持稳定错误码和回归测试；本地文件不直接对外提供下载；生产化前增加对象存储、病毒扫描、后台补偿任务和独立 Dataset 或 metadata 策略。
- 回退方案：如果 Dify 上传链路失败，保留现有查询 API 与预置 5 份文档；关闭文档路由不会影响认证、普通对话和 Agent。

## 九、可用于实习日记的素材

这一步让我理解到，文件上传并不是简单调用 `UploadFile.read()`。系统既要验证文件名、大小、MIME、文件头和解析器，还要处理本地文件、数据库任务和 Dify 异步索引之间无法使用同一事务的问题。选择保留失败台账、使用 UUID 存储路径并按需刷新状态，可以在教学复杂度和可审计性之间取得平衡。
