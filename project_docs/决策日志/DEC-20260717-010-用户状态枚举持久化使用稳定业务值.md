# DEC-20260717-010：用户状态枚举持久化使用稳定业务值

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：用户模型与初始数据库迁移
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

用户状态在 Python 中使用 `UserStatus.ACTIVE` 和 `UserStatus.DISABLED` 表达，但数据库字段和接口数据需要保存稳定的小写业务值 `active`、`disabled`。SQLAlchemy 的 `Enum` 默认可能持久化枚举成员名，即大写的 `ACTIVE`、`DISABLED`。

## 二、需要解决的问题

如果 ORM 写入大写成员名，而数据库默认值写入小写业务值，同一字段可能出现两套表示。读取由数据库默认值创建的数据时还可能触发枚举转换错误，影响用户登录、禁用和数据迁移。

## 三、可选方案

### 方案 A：数据库保存枚举成员名

- 优点：SQLAlchemy 默认行为，配置较少。
- 缺点：数据库值依赖 Python 命名方式；重命名成员可能破坏历史数据；与 API 常用的小写值不一致。

### 方案 B：数据库保存枚举业务值

- 优点：数据库、API 和业务语义统一；Python 成员名可在不改变存量数据的情况下调整。
- 缺点：需要显式配置 `values_callable`，并用测试防止后续误改。

## 四、最终决策

SQLAlchemy `Enum` 显式使用枚举成员的 `.value`，数据库只保存 `active` 和 `disabled`。数据库 `server_default` 使用 `active`，并增加模型元数据回归测试，断言迁移枚举值顺序和内容。

## 五、决策理由

业务值比 Python 代码中的符号名称更稳定。统一小写值可以减少 ORM、数据库和 API 之间的转换歧义，也能避免服务端默认值与 ORM 读取规则不一致。

## 六、影响与风险

- 正面影响：用户状态在各层表达一致，后续接口和迁移更容易理解。
- 潜在风险：未来增加状态时必须同时生成新迁移并更新测试。
- 回退方案：若状态模型复杂到需要附加属性，可改为独立状态表，但必须通过新决策和迁移完成。

## 七、验证证据

- 模型元数据测试确认枚举值为 `active`、`disabled`。
- Alembic 自动生成的迁移使用小写枚举值，服务端默认值为 `active`。
- 初始迁移升级、降级、再次升级均成功。
- `alembic check` 未检测到模型与数据库差异。
