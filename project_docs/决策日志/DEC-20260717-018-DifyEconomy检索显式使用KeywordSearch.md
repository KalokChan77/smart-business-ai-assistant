# DEC-20260717-018：Dify Economy 检索显式使用 Keyword Search

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Dify Dataset API、教学知识库 RAG 召回、后续 FastAPI Dify Gateway
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

教学知识库采用 Dify `economy` 索引，分段和关键词表均已完成。调用 `/v1/datasets/{dataset_id}/retrieve` 时，如果请求体只提供 `query`，Dify 1.15.0 会在数据库 `retrieval_model` 为空时使用语义检索默认值。economy 数据集没有对应的向量索引，因此接口可以返回 HTTP 200，但 `records` 为空。

在请求体显式提交 `search_method=keyword_search` 后，同一问题能够召回 5 条记录，并命中账户安全和订单退款文档。

## 二、需要解决的问题

1. 如何保证 economy 数据集始终走关键词检索；
2. 是否直接修改 Dify 数据库或上游源码；
3. 后续 FastAPI 代理层应如何构造稳定请求；
4. 如何避免把 HTTP 200 误判为 RAG 验收成功。

## 三、可选方案

### 方案 A：直接更新 Dify 数据库中的 `retrieval_model`

- 优点：客户端请求可以更短。
- 缺点：绕过 Dify 应用层，升级和迁移时难以追踪，教学项目不应直接修改平台内部数据。

### 方案 B：修改 Dify 上游源码，使 economy 自动默认关键词检索

- 优点：所有调用方自动生效。
- 缺点：维护上游补丁成本高，升级时容易冲突。

### 方案 C：调用方显式提交完整 `retrieval_model`

- 优点：不修改 Dify 内部数据和源码；请求行为可测试、可审计；未来可按索引类型切换策略。
- 缺点：FastAPI 客户端需要维护一份检索配置。

## 四、最终决策

- 当前 economy 教学知识库的所有检索请求显式提交 `retrieval_model`；
- 固定使用 `search_method=keyword_search`、`top_k=5`、关闭重排和分数阈值；
- 后续 FastAPI Dify Gateway 在服务端统一构造该参数，Vue3 前端只提交业务查询文本；
- 验收不仅检查 HTTP 状态，还必须检查 `records` 数量大于 0；
- 未来切换 `high_quality` 索引时，通过新决策日志评估语义检索或混合检索，不直接复用本配置。

## 五、决策理由

显式请求参数能把检索策略从 Dify 内部默认行为转化为本项目可见的接口契约。这样既不侵入 Dify 源码，也不需要直接修改平台数据库，并且能够在自动化测试中验证真正的召回结果。

## 六、影响与风险

- 正面影响：economy 知识库召回稳定；前端无需了解 Dify 检索模型结构；验收标准更准确。
- 潜在风险：如果知识库改为高质量向量索引，继续使用关键词检索会限制语义召回能力。
- 控制措施：检索配置集中在服务端；索引模式变化时建立新决策并增加回归测试。

## 七、验证证据

- 5 份文档均为 `completed`，共 26 个启用分段；
- 数据库关键词表包含 162 个关键词，并覆盖全部当前分段；
- 省略 `retrieval_model` 时接口为 HTTP 200、召回 0 条；
- 显式使用 `keyword_search` 后接口为 HTTP 200、召回 5 条；
- 召回结果包含账户安全和订单退款教学文档；
- 验证过程未输出任何 Dify API Key、Cookie、管理员密码或 Token。

## 九、可用于实习日记的素材

这次排查让我认识到，HTTP 200 只代表接口完成处理，不能代表知识库真正召回了内容。通过比较默认请求和显式关键词检索请求，我发现检索策略本身也是接口契约的一部分。把策略放在 FastAPI 服务端统一管理，可以减少前端复杂度，也能避免依赖第三方平台不明显的默认行为。
