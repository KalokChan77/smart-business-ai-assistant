## 一、背景

Dify 应用通常依赖外部大模型，但模型 API 还可能受到额度、网络、模型名称和账号权限影响。如果一开始就把平台部署、模型调用和业务逻辑混合测试，失败时很难判断问题来自哪一层。

## 二、需要解决的问题

需要先证明 Dify 的应用发布、Service API、鉴权、工作流执行和返回格式均正常，再接入真实大模型。

## 三、可选方案

### 方案 A：直接创建调用通义千问的完整问答应用

- 优点：更接近最终效果。
- 缺点：平台问题和模型问题耦合，排错困难，也会消耗模型额度。

### 方案 B：创建无模型依赖的回显 Chatflow 和 Workflow

- 优点：零模型费用、结果确定、测试速度快，能够独立验证 Dify Service API。
- 缺点：初始回答不具备真实智能能力。

## 四、最终决策

先创建两个无模型依赖应用：

- Chatflow：接收用户问题并返回“教学测试回复 + 原问题”；
- Workflow：接收 `query` 输入并原样输出。

完成 API 基线验收后，再把真实业务节点替换为通义千问、DeepSeek、知识检索或 LangGraph 编排。

## 五、决策理由

分层测试可以快速隔离故障。只要回显应用运行成功，就能证明 Dify 的部署、应用发布、API Key、HTTP 请求格式和反向代理正常，后续模型错误便可以集中在模型配置层排查。

## 六、影响与风险

- 正面影响：形成稳定、低成本、可重复执行的冒烟测试。
- 潜在风险：展示时若未替换节点，业务效果过于简单。
- 回退方案：保留回显应用作为系统健康检查，另建正式业务应用。

## 七、验证证据

- Chat API 返回 HTTP `200`，能够返回输入内容。
- Workflow API 返回 HTTP `200`，执行状态为 `succeeded`。
- Workflow 输出字段 `query` 与测试输入一致。

## 九、可用于实习日记的素材

我体会到测试复杂系统时应先建立最小可验证路径。先使用不依赖大模型的回显流程，可以避免同时处理多个不确定因素，这种分层排错方法也适用于后续 FastAPI 和 Vue3 开发。
