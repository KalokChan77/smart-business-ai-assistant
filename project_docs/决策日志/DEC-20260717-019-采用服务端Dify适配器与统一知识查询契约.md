# DEC-20260717-019：采用服务端 Dify 适配器与统一知识查询契约

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：FastAPI Dify Gateway、企业知识库查询、引用与安全拒答
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

本地 Dify 已完成部署，Chat、Workflow、Dataset 三类服务端 Key 均已通过不回显密钥的冒烟验证，目标 economy 知识库中的 5 份教学文档也已完成索引。下一步需要把 Dify 能力接入 FastAPI，使 Vue3 只依赖平台自己的接口，而不是直接接触 Dify Key、Dataset ID、检索模型和原始响应字段。

实际验证还发现，当前 Dify Chat 应用和 Workflow 是用于 Key 验证的教学占位流程：Chat 会返回带输入内容的测试回复，Workflow 会原样输出输入变量，尚不适合作为正式的知识答案生成器。因此，本阶段不能假设“Key 可用”等于“知识问答工作流已经配置完成”。

## 二、需要解决的问题

1. Dify HTTP 调用应放在哪一层，如何避免散落在 FastAPI 路由中；
2. economy 数据集的关键词检索参数由谁控制；
3. 平台知识查询接口应向前端返回哪些稳定字段；
4. 当前占位 Chat/Workflow 尚不能生成可靠答案时，如何完成不编造的最小验收闭环；
5. 空召回、提示词注入、上游认证失败、超时和协议异常如何安全处理；
6. 如何为以后替换成 Dify Chatflow、Workflow 或其他答案生成器保留低耦合边界。

## 三、可选方案

### 方案 A：Vue3 直接调用 Dify Service API

- 优点：开发代码最少，能快速看到 Dify 返回结果。
- 缺点：必须把 Dify Key 和 Dataset ID 暴露给浏览器；前端会依赖 Dify 原始协议；无法复用平台鉴权、请求追踪和统一错误处理。

### 方案 B：FastAPI 路由直接调用 Dify Chat 应用并原样透传

- 优点：比浏览器直连安全，调用链较短。
- 缺点：路由会同时承担 HTTP 适配、业务策略和协议转换；当前 Chat 应用只是占位回复，且没有返回知识库引用；Dify 升级会直接影响前端。

### 方案 C：建立 Dify 适配层和 Knowledge 业务层，当前先采用可追溯的抽取式答案

- 内容：`app/ai/dify/` 只处理 Dify Dataset HTTP 协议和安全错误；`app/knowledge/` 负责鉴权后的查询用例、空召回、安全拒答、答案组织和平台响应模型。
- 优点：职责清晰；Key 只在服务端；检索策略可测试；即使占位 Chat/Workflow 尚未配置，也能基于真实召回片段返回不编造、带引用的结果；以后可替换答案生成器而不改变前端契约。
- 缺点：首版答案主要由知识片段抽取组织，语言自然度不如配置完成的生成式 RAG。

## 四、最终决策

采用方案 C，并固定以下边界：

1. 新建 `app/ai/dify/`，只负责 Dify HTTP 请求、响应解析和上游错误分类，不依赖 FastAPI 路由或业务用户模型；
2. 新建 `app/knowledge/`，负责知识查询业务、输入安全策略、统一响应、错误映射和认证路由；
3. 对外接口固定为 `POST /api/v1/knowledge/query`，请求体首版只接收业务字段 `query`；
4. Dify Dataset ID、API Key、`retrieval_model` 均由 FastAPI 服务端提供，Vue3 不得提交或接收这些字段；
5. economy 检索继续显式使用 `keyword_search`、`top_k=5`，关闭重排和分数阈值；
6. 返回平台统一的 `outcome`、`answer`、`citations` 和 `retrieval_count`，不透传 Dify 原始对象；引用只公开排序、文档名、片段和公开相关度，不公开 Dataset ID；
7. 当前阶段采用抽取式答案：只组织 Dify 已召回的片段，不调用占位 Chat/Workflow 生成答案；空召回返回固定的“依据不足”说明；
8. 对索取系统提示词、Key、Token、Cookie、密码或内部配置的明显注入请求，返回固定安全拒答并记录安全事件，不把原始问题写入日志；
9. Dify 401/403、429、5xx、网络失败、超时和协议异常全部转换成稳定且脱敏的平台错误；上游响应正文不得进入客户端错误信息；
10. 后续配置好正式 Dify Chatflow/Workflow 后，通过新的答案生成器接口替换抽取式实现，保持 `/knowledge/query` 响应契约不变；
11. 文档上传、索引任务、数据库文档模型和会话/AI Run 持久化作为后续子阶段单独决策，避免本次网关接入同时引入文件存储和数据库迁移风险。

## 五、决策理由

该方案符合“浏览器只访问 FastAPI、密钥只保存在服务端”的安全基线，也符合路由薄、业务服务高内聚、第三方适配器低耦合的软件工程要求。当前先用抽取式答案，可以确保每一句内容都来自真实召回片段，避免占位应用回显提示词或生成无依据内容，同时仍能完成认证、检索、引用、空答案和错误映射的可测闭环。

把答案生成器视为可替换组件后，后续无论使用 Dify Chatflow、Workflow、DeepSeek 还是百炼，都不需要让 Vue3 改用第三方协议，也不会建立 LangGraph 与 Dify 的循环调用。

## 六、影响与风险

- 正面影响：Dify Key 不出服务端；前端契约稳定；检索策略集中；引用与空召回可重复测试；第三方错误不会泄露内部响应。
- 潜在风险：抽取式答案可能包含多个相关片段，表达不如大模型自然；economy 关键词检索的 `score` 不能作为可靠语义阈值；当前知识库为所有已登录用户共享。
- 控制措施：回答明确标记为“根据当前知识库找到的相关说明”；保留完整 citation；不使用当前 `score=0.0` 判定无答案，而以 Dify `records` 是否为空为准；正式生成式工作流、租户级知识库和权限细化另立决策。
- 回退方案：若适配器接入失败，可暂时保留现有 `verify_dify_flow.py` 作为独立验收，不影响已经完成的普通对话和 LangGraph Agent 路由。

## 七、验证证据

- 三类 Dify Key 均已配置且通过服务端冒烟验证，未回显任何密钥；
- Chat API 返回 HTTP 200，但当前输出表现为教学占位回复；
- Workflow API 执行状态为 `succeeded`，但当前输出表现为输入变量透传；
- Dataset API 可访问目标数据集，5 份文档均为 `completed`；
- 显式 economy 关键词检索返回 HTTP 200 和 5 条记录；
- “2028 年上市计划”测试问题返回 0 条记录，适合验证无依据分支；
- Dify Client、惰性 Dataset Retriever、Knowledge Service、安全策略和认证路由已经实现；
- 首轮审查发现“安全拒答被 Dify 配置依赖阻断”、额外字段被静默忽略、去空白与长度校验顺序不一致、内部配置拒答范围不足 4 个中等问题，均完成根因修复和回归测试；
- Dify 配置和 HTTP Client 现在只在真实检索时校验和创建，安全拒答不依赖 Dify 是否可用；
- 知识查询请求拒绝所有未声明字段，禁止前端夹带 Key、Dataset ID、`retrieval_model` 等内部参数；
- 定向测试 47 项通过，全量测试 120 项通过；
- 真实认证 API 冒烟覆盖 `answered`、`no_match`、`refused` 三条路径，均为 HTTP 200，命中路径返回 5 条 citation；
- 临时用户、角色和 Redis 吊销键在冒烟结束后均清理为 0；
- `compileall`、`pip check`、`alembic check`、Dify 三类 Key 验证、健康检查和就绪检查全部通过；
- 修改范围与当前环境变量中真实 Secret 的字面值扫描结果为 0；
- 复审未发现新的 P1/P2 问题；当前虚拟环境未安装 `ruff`、`mypy` 或 `pyright`，因此使用全量测试、AST/编译检查和人工审查作为现阶段验证证据。

## 九、可用于实习日记的素材

这次配置完成后的验证让我认识到，API Key 可用只说明认证和网络正常，并不能证明应用流程已经具备正式业务能力。Chat 和 Workflow 都能返回成功状态，但实际还是占位回显，因此需要把“服务连通”“知识召回”“答案生成”拆成不同验收层。通过 Dify 适配器与 Knowledge 业务层分离，首版可以先用真实召回片段完成不编造的引用回答，再在不改变前端接口的前提下逐步替换生成器。
