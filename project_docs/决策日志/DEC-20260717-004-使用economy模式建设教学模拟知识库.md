## 一、背景

项目需要支持 PDF、Word 和其他业务文档上传，并通过 RAG 回答企业、产品、订单退款、文件规范、账户安全等问题。当前阶段首先要验证知识库 API 和文档索引流程，不要求追求大规模向量检索效果。

## 二、需要解决的问题

需要确定第一版知识库采用高质量向量索引还是 economy 关键词索引，并准备可控的模拟业务文档进行测试。

## 三、可选方案

### 方案 A：high_quality 向量索引

- 优点：语义检索能力更强，更接近正式 RAG 应用。
- 缺点：依赖 Embedding 模型配置和调用额度，部署变量更多。

### 方案 B：economy 关键词索引

- 优点：不依赖 Embedding 模型，可快速完成文档切分、索引和检索 API 验收。
- 缺点：语义匹配能力有限，后续需要升级。

## 四、最终决策

第一阶段创建 economy 模式的“智慧商务教学模拟知识库”，上传 5 份模拟业务文档，先完成 Dataset Key、文档上传、异步索引和检索接口验收。

## 五、决策理由

本阶段重点是证明 RAG 全流程可运行，而不是比较模型效果。economy 模式可以减少模型依赖和额度消耗，使问题定位更清晰。后续获得正式资料和稳定 Embedding 模型后，再迁移到 high_quality。

## 六、影响与风险

- 正面影响：知识库可以在无 Embedding 模型的情况下完成测试。
- 潜在风险：检索结果分数和语义相关度不如向量检索。
- 回退方案：保留原始文档，后续新建 high_quality 数据集或重建索引。

## 七、验证证据

- Dataset API 返回 HTTP `200`。
- 5 份模拟文档全部完成索引，结果为 `5 completed / 0 error`。
- 检索问题“退款申请需要满足什么条件？”返回 HTTP `200`，命中订单退款与账户安全文档。

我理解了 RAG 不只是“上传文件”，还包括文件存储、异步任务、文本切分、索引和检索。先选择 economy 模式能够降低初期复杂度，是一种先保证流程可用、再逐步提升效果的工程方法。
