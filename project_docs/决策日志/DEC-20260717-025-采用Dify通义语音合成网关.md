# DEC-20260717-025：采用 Dify 通义语音合成网关

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Dify TTS 配置、FastAPI 文字转语音接口、前端语音播放能力
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

需求基线把文字转语音列为核心交付能力，当前系统已经完成 Dify 本地部署、通义模型供应商配置、三类 Dify Service API Key 配置以及 FastAPI 鉴权体系，但 Dify 应用“智慧商务教学问答”的文字转语音功能仍处于关闭状态，FastAPI 也尚未提供平台统一的语音接口。

Dify 本地环境已经安装通义插件并能够发现 `qwen3-tts-flash` 模型。该模型单段文本限制为 512 字符，支持中文语言标识 `zh-Hans`，默认声音为 `Cherry`。项目只要求本地教学演示与功能测试成功，不需要建设大规模音频任务队列或长期音频存储。

## 二、需要解决的问题

1. 前端直接调用 Dify，还是统一通过 FastAPI 调用；
2. TTS 模型、声音和语言由前端自由指定，还是由服务端统一管理；
3. 如何避免在浏览器、响应、日志和文档中暴露 Dify App Key；
4. 如何限制输入文本和上游音频响应大小，防止异常请求占用过多内存；
5. Dify 的认证失败、限流、超时、服务异常和错误响应如何转换为平台统一错误；
6. 音频是否需要写入数据库或本地文件；
7. 如何为 Dify 提供稳定用户标识，同时避免直接传递平台用户 UUID；
8. 如何保留以后替换语音供应商的能力，而不让路由和业务服务依赖 Dify 私有协议。

## 三、可选方案

### 方案 A：Vue3 前端直接调用 Dify TTS

- 优点：开发步骤少，音频可以直接返回浏览器。
- 缺点：必须把 Dify App Key 暴露给浏览器；绕过平台鉴权、审计和统一错误处理；前端会与 Dify 私有接口强耦合。

### 方案 B：FastAPI 直接调用阿里云百炼 TTS

- 优点：调用链较短，可以完全自行控制模型参数。
- 缺点：重复建设 Dify 已经具备的模型供应商和凭据管理；平台同时维护两套通义调用配置；不利于展示 Dify 低代码平台在项目中的实际作用。

### 方案 C：FastAPI 作为安全网关调用 Dify Service API

- 内容：Dify 负责管理默认 TTS 模型、声音和语言，FastAPI 负责平台鉴权、输入限制、稳定用户标识、响应大小限制、错误映射和安全响应头。
- 优点：浏览器不接触任何上游 Key；复用现有 Dify 通义插件；职责边界清晰；前端只依赖平台 API；以后可以通过端口替换实现。
- 缺点：增加一次本地 HTTP 调用，需要同时保证 FastAPI 和 Dify 可用。

### 方案 D：生成音频后持久化文件和数据库记录

- 优点：可以重复播放和统计音频生成历史。
- 缺点：需要处理文件生命周期、存储配额、用户删除和隐私清理；超出当前教学验收范围。

### 方案 E：同步生成并只在响应中返回音频

- 优点：实现简单，不留下音频正文；适合短文本和本地演示；减少隐私与存储风险。
- 缺点：相同文本重复播放会再次消耗模型额度，长文本不适合该模式。

## 四、最终决策

采用方案 C 与方案 E，并固定以下规则：

1. Dify 工作空间默认 TTS 模型配置为 `langgenius/tongyi/tongyi` 的 `qwen3-tts-flash`；
2. Dify 应用“智慧商务教学问答”启用文字转语音，默认声音为 `Cherry`，语言为 `zh-Hans`；
3. FastAPI 新增独立 `app/audio/` 领域，并提供 `POST /api/v1/audio/tts`；
4. 任意已登录角色可以调用该接口，未登录请求返回 401；
5. 前端请求只允许提交 `text`，不得提交 Dify Key、Provider、模型名、声音、语言、内部 App ID、`message_id` 或其他额外字段；
6. 文本去除首尾空白后必须为 1～500 个字符。500 字符为模型 512 字符上限预留协议和后续适配余量；
7. FastAPI 根据租户 ID 和用户 ID 生成不可逆、稳定的 SHA-256 用户别名传给 Dify，不向 Dify 直接暴露平台 UUID；
8. Dify Key 只从服务端 `SecretStr` 配置读取，HTTP Client 使用 `trust_env=False`，避免本机代理干扰本地 Dify 调用；
9. FastAPI 不盲信 Dify 声明的媒体类型，而是根据受支持的文件签名识别 MP3 或 WAV，并分别返回 `audio/mpeg` 或 `audio/wav`；成功响应携带 `Cache-Control: no-store`、`Pragma: no-cache` 和 `X-Content-Type-Options: nosniff`；
10. FastAPI 以流式方式读取上游响应并设置 5 MiB 上限；空响应、无法识别的音频响应和超限响应均视为上游协议错误；V0.1 不额外引入 FFmpeg 或音频转码依赖；
11. Dify 认证、限流、超时、不可用、请求拒绝和协议错误转换为平台统一、脱敏的 502/503/504 错误，禁止透传上游正文；
12. V0.1 不持久化音频正文、不写临时音频文件、不提供模型和声音自由选择、不实现批量或异步长文本合成；
13. 业务服务只依赖语音网关端口，Dify Client 负责私有 HTTP 协议，路由只负责鉴权、请求绑定和二进制响应。

## 五、决策理由

1. **安全性**：Dify App Key 留在服务端，浏览器无法读取；上游错误正文不会进入平台响应；
2. **高内聚低耦合**：语音 Schema、端口、Dify Client、服务、依赖和路由位于独立领域，其他模块不需要理解 TTS 私有协议；
3. **教学价值**：能够同时展示 FastAPI 网关设计、Dify 低代码模型管理和阿里云通义语音模型调用；
4. **可靠性**：输入长度、响应类型、响应大小、超时和状态码均有明确边界；
5. **隐私与成本**：只同步返回音频，不保存正文或音频文件，减少数据生命周期与存储管理成本；
6. **可替换性**：以后如改为直接调用百炼或其他供应商，只需实现相同语音网关端口，无需修改路由契约。

## 六、影响与风险

- 正面影响：前端可以通过统一平台接口播放 AI 回答；Dify TTS 能力进入真实业务链路；凭据和模型配置继续集中在服务端。
- 潜在风险：上游模型额度不足、网络超时或 Dify 配置被修改会导致生成失败；同步接口会占用一次请求连接；500 字符以上文本需要前端或未来服务分段；不同版本 Dify 插件可能返回不同音频容器，因此必须保留文件签名回归测试。
- 回退方案：关闭 FastAPI 音频路由或在 Dify 中关闭 TTS 即可停止功能；恢复配置前保存的工作流特征和默认模型记录可用于回滚；不涉及业务数据库迁移和音频数据清理。

- 配置前 `GET /v1/parameters`：HTTP 200，顶层 `text_to_speech.enabled=false`；
- Dify 应用匹配数：1，模式为 `advanced-chat`，Service API 已启用；
- Dify 草稿工作流存在，当前已发布工作流存在；
- 通义 TTS 模型：已发现并处于可用状态；
- 配置脚本执行成功，草稿与已发布工作流的 TTS 均已启用，声音和语言已配置；
- 配置脚本再次执行时输出 `published_new_version=false`，证明目标状态下不会重复创建已发布工作流版本；
- 配置后 `GET /v1/parameters`：HTTP 200，顶层 `text_to_speech.enabled=true`；
- 真实 `POST /v1/text-to-audio`：HTTP 200，响应体非空；上游头声明 `audio/mpeg`，实际文件签名为 RIFF/WAVE，并通过 Python `wave` 模块解析为 24 kHz、16 bit、单声道有效音频；
- 上述差异证明不能只校验响应头，因此 V0.1 改为识别 MP3/WAV 文件签名并返回真实媒体类型；
- TTS 配置、客户端、工厂、服务和 API 共收集 50 项定向测试并全部通过，覆盖四类已登录角色、严格字段校验、状态码映射、超时、网络异常、非音频、空响应、无效 WAV、伪 ID3、不完整 MPEG 帧、MP3/WAV 识别和不可放宽的 5 MiB 边界；
- 后端共收集 263 项测试并全部通过，`compileall`、`pip check` 和 `alembic check` 均通过，当前迁移仍为 `2b67c8e7c1ea (head)`，没有结构漂移；
- OpenAPI 已包含 `/api/v1/audio/tts`，应用版本更新为 `0.8.0`，成功响应声明 `audio/mpeg` 与 `audio/wav`；
- 真实 FastAPI 冒烟完成登录、匿名 401、额外字段 422、TTS 200、WAV 解析、安全响应头和退出验证；临时用户、角色及 Redis 测试键最终均为 0；
- 独立代码审查初次发现“上限可放宽到 25 MiB、MP3 只检查 ID3 前缀、冒烟脚本可能跳过 Token 清理证据”3 个 MEDIUM，以及决策日志状态滞后 1 个 LOW；整改后再次复核，全部问题均已关闭，最终结论为 `APPROVE`，无剩余 HIGH 或 MEDIUM；
- 所有验证输出只记录布尔状态、HTTP 状态、媒体类型、字节数和测试数量，不输出任何 Key 或上游错误正文。

## 九、可用于实习日记的素材

本阶段从“Dify 已经安装模型”与“业务接口真正可用”的差异出发，先核验应用参数，再设计 FastAPI 安全网关。主要收获是第三方能力不能直接等同于平台能力：还需要补齐鉴权、输入边界、凭据隔离、错误映射、响应大小限制和验证闭环。通过把模型配置留在 Dify、把业务安全边界放在 FastAPI，可以同时发挥低代码平台和后端工程的优势。
