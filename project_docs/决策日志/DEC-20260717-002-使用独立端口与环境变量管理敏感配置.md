# DEC-20260717-002：使用独立端口与环境变量管理敏感配置

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Dify 端口规划与密钥管理
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

开发机已有服务占用 `5432` 和 `8080` 端口，同时项目需要保存百炼、DeepSeek 和 Dify 的多类 API Key。若直接使用默认端口或把密钥写入代码，容易产生端口冲突和安全风险。

## 二、需要解决的问题

需要制定不会与现有服务冲突的端口方案，并确定密钥、管理员密码和服务地址的统一管理方式。

## 三、可选方案

### 方案 A：停止现有服务并使用默认端口

- 优点：配置接近官方默认值。
- 缺点：会影响其他项目，容易造成数据或服务中断。

### 方案 B：为 Dify 分配独立端口并使用环境变量

- 优点：多个项目可以并行运行，配置集中，可通过 `.gitignore` 防止密钥进入版本库。
- 缺点：文档和调用方必须使用新的端口地址。

## 四、最终决策

- Dify HTTP 控制台端口使用 `18080`；
- HTTPS 预留端口使用 `18443`；
- 插件调试端口使用 `15003`；
- Dify Service API 地址使用 `http://localhost:18080/v1`；
- 项目 API Key 统一写入根目录 `.env`；
- Dify 管理员信息单独写入 `dify-self-host/.admin.env`；
- 所有敏感环境文件权限设置为 `0600`，并禁止进入 Git。

## 五、决策理由

独立端口不会干扰已有 PostgreSQL 和其他 Web 服务。环境变量方式符合 FastAPI、Docker Compose 和第三方 SDK 的常见配置方式，也便于之后切换本地、测试服务器和 Oracle Cloud 环境。

## 六、影响与风险

- 正面影响：降低端口冲突和密钥泄露风险。
- 潜在风险：开发人员可能误用默认的 `8080` 地址。
- 回退方案：所有模块统一从 `DIFY_BASE_URL` 读取地址，不在代码中硬编码端口。

## 七、验证证据

- 根目录 `.env` 权限为 `0600`。
- Dify Docker `.env` 权限为 `0600`。
- `DIFY_CHAT_APP_API_KEY`、`DIFY_WORKFLOW_API_KEY`、`DIFY_DATASET_API_KEY` 和 `DIFY_DATASET_ID` 均已配置。
- 日志和文档中未记录任何明文密钥。
