# DEC-20260718-028：固定前端 TypeScript 版本并采用锁文件保障可复现构建

- 日期：2026-07-18
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Vue3 前端工程可复现安装、类型检查与生产构建
- 关联实习日记：`project_docs/实习日记/2026-07-18-实习日记.md`
- 关联决策：补充 `DEC-20260717-027`

## 一、背景

前端采用 Vue 3 + TypeScript + Vite + `vue-tsc` 作为类型检查与构建链路。安装过程中若直接拉取最新 TypeScript 主版本，可能与当前 `vue-tsc` 的类型系统假设不一致，导致 `typecheck` 或 `build` 失败。教学演示项目需要保证任意成员在本地按文档安装后结果一致。

## 二、需要解决的问题

1. TypeScript 主版本升级是否允许直接跟随 npm 最新标签；
2. 如何保证开发机、答辩演示机和生产构建使用同一组依赖；
3. 如何在不引入过重前端基建的前提下提高可复现性。

## 三、可选方案

### 方案 A：不固定 TypeScript，始终使用最新版本

- 优点：可尽早体验新特性。
- 缺点：与 `vue-tsc` 不兼容时会直接阻断类型检查和构建；不同机器结果不一致。

### 方案 B：固定 TypeScript 到与 `vue-tsc` 兼容的 5.x 版本，并提交 `package-lock.json`

- 优点：安装结果可复现；类型检查稳定；符合教学演示“能稳定跑起来”的目标。
- 缺点：需要在升级 `vue-tsc` 或 TypeScript 时做显式验证。

### 方案 C：放弃 TypeScript，改用纯 JavaScript

- 优点：安装链路更简单。
- 缺点：与当前后端严格契约、SSE 事件和多角色页面类型安全性冲突，不利于后续维护。

## 四、最终决策

采用方案 B：

1. 将前端 `typescript` 固定为 `~5.9.3`，与当前 `vue-tsc ^3.3.7` 兼容；
2. 提交 `frontend/package-lock.json`，安装时优先使用锁文件；
3. 前端标准验证命令固定为：
   - `npm run typecheck`
   - `npm test`
   - `npm run build`
   - `npm audit --omit=dev`
4. 升级 TypeScript 或 `vue-tsc` 时，必须先跑通上述命令，再单独写决策日志。

## 五、决策理由

1. **可复现**：锁文件避免“我这边能跑、你那边失败”；
2. **教学稳定**：集中实训时间短，不应把时间消耗在工具链漂移上；
3. **契约优先**：类型检查能尽早发现前后端字段不一致，例如回复建议状态和百分比单位；
4. **成本可控**：只需固定版本，不需要额外 monorepo 工具。

## 六、影响与风险

- 正面影响：类型检查、测试和构建结果稳定，便于答辩前快速恢复环境；
- 潜在风险：未来 `vue-tsc` 升级可能要求同步调整 TypeScript 版本；
- 回退方案：若必须升级，可在通过完整验证后更新 `package.json` 与锁文件，并新增替代决策日志。

## 七、验证证据

- `frontend/package.json` 中 `typescript` 为 `~5.9.3`，`vue-tsc` 为 `^3.3.7`；
- 锁文件记录 TypeScript `5.9.3`、`vue-tsc` `3.3.7`；
- `npm run typecheck` 通过；
- `npm test`：7 个测试文件、21 项测试通过；
- `npm run build` 生产构建成功；
- `npm audit --omit=dev`：0 vulnerabilities。
