# DEC-20260717-013：采用租户内 RBAC 与仓储服务分层

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：用户管理、当前用户依赖、角色权限校验
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

用户和角色模型已经包含 `tenant_id`。技术方案要求支持企业用户、客服、管理员和决策者等角色，同时规定 API 路由只负责参数校验和调用应用服务，不能直接编写数据库查询。

## 二、需要解决的问题

需要确定权限作用域、当前用户在各层的表达方式，以及路由、服务和 SQLAlchemy 之间的边界。若直接在每个路由查询用户和角色，后续会话、AI 和知识库模块会重复认证逻辑并增加越权风险。

## 三、可选方案

### 方案 A：路由直接查询 ORM 并判断角色

- 优点：文件较少，短期开发快。
- 缺点：路由与 SQLAlchemy 强耦合，权限判断重复，难以测试和复用。

### 方案 B：只相信 JWT 内的角色声明

- 优点：请求时少一次数据库查询。
- 缺点：管理员修改角色或禁用用户后，旧令牌在过期前仍保留旧权限。

### 方案 C：仓储 + 应用服务 + 当前主体依赖

- 优点：SQL 查询集中在仓储；业务规则集中在服务；路由只处理 HTTP；每次认证使用数据库中的当前状态和角色作为权威来源。
- 缺点：模块数量增加，需要明确对象和事务边界。

## 四、最终决策

- 用户和角色权限限定在当前 `tenant_id` 内，管理员不能通过普通用户管理 API 跨租户操作；
- SQLAlchemy 查询集中在 `users/repository.py`；
- 用户创建、修改、唯一性和角色规则集中在 `users/service.py`；
- 认证流程集中在 `auth/service.py`，密码与 JWT 放在 `auth/security.py`；
- 路由通过依赖获取不可变的当前主体对象，不把 ORM 实体直接传播到其他领域；
- JWT 可以携带角色快照供客户端显示，但服务端授权以数据库中的当前用户状态和角色为准；
- 第一阶段内置角色代码为 `admin`、`customer_service`、`user` 和 `decision_maker`；
- 使用独立的交互式初始化脚本创建首个租户管理员，避免开放无认证的“创建首个管理员”HTTP 接口。

## 五、决策理由

仓储隔离数据库细节，服务集中业务规则，当前主体隔离认证和 ORM 生命周期，符合高内聚低耦合要求。数据库作为权限权威来源，可以让禁用用户和角色变更更快生效。交互式初始化脚本既方便教学演示，也避免留下公开注册管理员的高风险接口。

## 六、影响与风险

- 正面影响：权限边界统一，后续会话、AI、知识库和管理接口可复用同一主体依赖。
- 潜在风险：每个受保护请求需要查询数据库并检查 Redis，增加少量开销。
- 回退方案：后续可用短 TTL 缓存当前主体，但缓存失效策略必须另写决策日志，不改变数据库权威来源。

## 七、验证证据

- 自动化 API 测试证明未登录返回 HTTP 401、非管理员返回 HTTP 403，管理员可以查询、创建和修改本租户用户。
- 真实端到端验收使用两个临时租户，租户 A 管理员修改租户 B 用户返回 HTTP 404 `user_not_found`。
- 临时管理员登录后被移除 `admin` 角色，旧 JWT 再访问 `/users` 立即返回 HTTP 403，证明数据库角色覆盖 JWT 快照。
- 用户被禁用后，旧访问令牌访问 `/auth/me` 立即返回 HTTP 403 `user_disabled`。
- 真实 PostgreSQL 集成测试发现并修复提交后 `updated_at` 过期导致的 `MissingGreenlet`，仓储现会显式刷新标量字段和角色关系。
- `auth/router.py` 与 `users/router.py` 未包含 SQLAlchemy 查询和事务提交，SQL 集中在仓储层。
- 交互式 `scripts.bootstrap_admin` 只输出租户编号和用户名，不输出密码。

## 九、可用于实习日记的素材

我理解到 JWT 中携带角色并不代表服务端必须永远相信这个快照。为了让禁用和权限调整及时生效，可以在认证依赖中重新读取数据库状态，并把当前用户转换成与 ORM 解耦的主体对象供其他业务模块使用。
