# DEC-20260717-011：采用纯 ASGI 请求上下文与统一错误契约

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：统一异常响应、请求 ID、结构化日志
- 关联契约：`project_docs/API通用响应契约-V0.1.md`
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

技术方案要求 FastAPI 统一承担异常处理、审计和日志能力，后续 AI 对话接口还需要使用 SSE 长连接。如果在每个路由中分别生成请求编号、捕获异常和打印日志，会产生重复代码，并使不同模块返回不同格式。

## 二、需要解决的问题

需要建立一个不依赖具体业务领域的 HTTP 横切层，使普通 REST、错误响应和未来 SSE 都能使用同一个请求 ID；同时保证内部异常不会把堆栈、数据库信息或密钥返回给浏览器。

## 三、可选方案

### 方案 A：在每个路由中手动处理

- 优点：单个接口容易理解。
- 缺点：代码重复，容易漏加请求 ID；错误格式和日志字段难以保持一致；路由承担了不属于业务的职责。

### 方案 B：使用 `BaseHTTPMiddleware`

- 优点：写法接近普通请求函数，入门简单。
- 缺点：对流式响应和上下文变量存在额外限制，不适合作为未来 SSE 的长期基础。

### 方案 C：纯 ASGI 中间件 + 集中异常处理器

- 优点：兼容流式响应；不需要新增依赖；可以在 ASGI 响应开始时统一注入请求 ID；业务异常、参数校验和 HTTP 异常可以集中映射。
- 缺点：实现比函数式中间件稍复杂，需要正确处理“响应已经开始后发生异常”的情况。

## 四、最终决策

- 使用纯 ASGI 中间件管理请求 ID、请求耗时和访问日志；
- 客户端提供的 `X-Request-ID` 只有在满足长度和字符白名单时才接受，否则生成新的 UUID；
- 使用 `ContextVar` 在同一异步请求链中传递请求 ID；
- 统一错误响应格式为 `error` 对象加顶层 `request_id`；
- 分别处理应用业务异常、FastAPI 参数校验异常和 HTTP 异常；
- 未知异常只向客户端返回通用错误信息，详细堆栈仅写入服务端日志；
- 日志采用标准库 `logging` 输出 JSON，不为基础日志功能新增第三方依赖；
- 不记录请求正文、Token、Cookie 和 API Key。

## 五、决策理由

横切能力集中在 `app/core/` 可以保持业务领域高内聚，路由只负责参数校验和调用服务。纯 ASGI 中间件不会依赖特定业务模块，也能为未来 SSE 保留正确的流式响应边界。统一请求 ID 可以把前端报错、API 响应和服务端日志关联起来，提高排错效率。

## 六、影响与风险

- 正面影响：所有接口拥有一致的错误结构和追踪编号，后续模块无需重复实现。
- 潜在风险：流式响应已经发送响应头后发生异常，不能再改写为普通 JSON 500。
- 回退方案：此类流式异常记录为失败日志并终止流；未来 SSE 协议使用专门的 `error` 事件通知客户端。

## 七、验证证据

- 横切基础能力完成时自动化测试共 18 项通过，覆盖合法和非法请求 ID、404、405、参数校验错误、业务异常、未知异常、正常流式响应和响应开始后的流式异常。
- 测试确认所有错误响应的 `request_id` 与 `X-Request-ID` 响应头一致。
- 未知异常和手动抛出的 5xx HTTP 异常均返回固定 `internal_server_error`，响应中不包含 Python 堆栈、数据库地址和内部异常原文。
- JSON Formatter 测试确认日志可被 JSON 解析，并包含时间、级别、消息、请求 ID、方法、路径、状态码和耗时；常见 API Key、Token、密码和 URL 用户信息会被替换为 `[REDACTED]`。
- 中间件集成测试确认每个完成的请求只生成一条结构化访问记录。
- 流式响应测试确认 `text/event-stream` 正文未被中间件缓存或改写。
- 真实 Uvicorn 启动日志与访问日志均为 JSON；健康检查和 readiness 返回 HTTP 200，404 返回统一错误结构。
- OpenAPI 版本升级为 `0.3.0`。
