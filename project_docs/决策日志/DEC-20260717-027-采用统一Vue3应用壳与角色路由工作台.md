# DEC-20260717-027：采用统一 Vue3 应用壳与角色路由工作台

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Vue3 登录、会话恢复、四角色工作台、核心业务页面与前端测试
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

需求基线把 Vue3 登录、角色工作台和基础管理页面列为核心交付，并要求浏览器只访问 Vue3 与 FastAPI，不直接调用 DeepSeek、LangGraph、Dify 或携带任何模型和低代码平台密钥。当前 FastAPI 已完成认证、用户、会话、AI SSE、知识查询、知识文档、客服辅助、语音合成和基础统计接口，但工作区尚无独立业务前端。

项目用于本地教学演示，第一版需要覆盖 `admin`、`decision_maker`、`customer_service`、`user` 四类角色，既要展示完整业务闭环，也要控制依赖、页面数量和部署复杂度。因此需要先统一鉴权、路由、错误、SSE 和页面状态设计，再分别接入角色页面，避免四套门户重复实现。

## 二、需要解决的问题

1. 四类角色是否分别建立独立应用，还是共享一个应用壳；
2. 登录后如何根据数据库中的当前角色选择默认工作台；
3. 访问令牌和刷新令牌在当前后端契约下如何保存与轮换；
4. 如何保证侧栏、路由守卫和后端 RBAC 的职责边界；
5. 前端是否直接依赖 Axios、UI 组件库和图表库；
6. SSE、普通 JSON、文件上传和音频 Blob 如何使用统一请求基础设施；
7. 本地开发如何避免额外配置 CORS；
8. 第一版哪些页面必须真实接入，哪些管理能力暂不展示；
9. 如何为加载、空数据、业务无答案、权限失败和上游不可用提供稳定状态；
10. 如何验证四角色分流、权限导航和关键交互，而不在测试中写入真实凭据。

## 三、可选方案

### 方案 A：四套独立 Vue 应用

- 优点：角色入口直观，部署后可分别定制。
- 缺点：登录、请求层、错误处理、会话、知识查询和布局大量重复；本地演示与维护成本过高。

### 方案 B：统一 Vue3 应用壳与 RBAC 动态导航

- 优点：公共鉴权、请求、SSE、布局和状态组件可复用；四角色仍可拥有明确工作台；便于本地切换账号演示。
- 缺点：需要严格维护路由角色表、默认落地页和多角色选择规则。

### 方案 C：单页演示大屏

- 优点：开发速度快，所有功能集中展示。
- 缺点：不能真实验证角色权限、页面路由和业务流程，不符合需求中的角色工作台结构。

### 方案 D：使用大型 UI/图表框架快速拼装

- 优点：表格、表单和图表组件丰富。
- 缺点：当前范围会增加包体、样式覆盖和学习成本；基础指标可用 CSS 与 SVG 完成，不需要先引入额外框架。

## 四、最终决策

采用方案 B，并固定以下规则：

1. 在根目录新增独立 `frontend/`，采用 Vue 3、TypeScript、Vite、Vue Router、Pinia、Vitest 与 Vue Test Utils；
2. 不引入 Axios、UI 组件库和图表库。普通请求统一使用浏览器 `fetch`，统计图使用可访问的 CSS/SVG 与数值表，后续确有复杂图表需求时再单独决策；
3. 使用统一应用壳：左侧动态侧栏、顶部用户与工作台信息、主内容路由出口；小于 768px 时侧栏可收起并允许表格横向滚动；
4. 登录页固定为 `/login`。成功获得令牌后立即调用 `/api/v1/auth/me`，不只根据 JWT 内容决定角色；
5. 当前后端通过响应体返回双 JWT，因此第一版将令牌对保存在 `sessionStorage`，仅在当前浏览器标签会话中恢复；退出、刷新失败和 401 时立即清理。该方案不等同于生产级 HttpOnly Cookie，后续公网部署需重新评估；
6. 请求层在受保护请求收到 401 后只允许执行一次去重的刷新令牌请求，并且最多重放原请求一次，防止并发重复刷新和无限循环；
7. 侧栏和路由守卫使用同一份角色元数据生成，但前端隐藏页面只改善体验，最终权限始终以后端 401/403/404 为准；
8. 多角色默认工作台顺序固定为 `admin -> decision_maker -> customer_service -> user`，并允许用户在自己拥有的工作台之间切换；
9. 四类角色默认落地页固定为：
   - `admin`：`/admin/overview`；
   - `decision_maker`：`/decision/overview`；
   - `customer_service`：`/service/tickets`；
   - `user`：`/app/chat`；
10. 第一版真实接入以下主线：
    - 企业用户：AI SSE 对话、知识问答、我的工单、历史会话、反馈与 TTS；
    - 客服人员：工单列表/详情、AI 分类、建议回复和人工确认；
    - 管理员：用户管理、知识文档上传/任务状态、统计入口；
    - 决策者：总览、咨询趋势、分类、满意度和 AI 质量；
11. 模型配置、Prompt 管理和审计日志当前没有已实现公开 API，不在第一版创建伪功能页面；
12. Vite 开发服务器把 `/api` 代理到可配置的本地 FastAPI 地址，浏览器仍使用相对 `/api/v1` 路径，避免写死域名并避免为本地开发扩大 CORS；
13. 普通错误统一解析平台 `error.code`、`error.message` 和 `request_id`；401 清理会话并返回登录页，403 进入无权限页，404 使用不泄露跨租户资源存在性的提示，409/422 保留页内业务提示；
14. 知识查询的 `no_match` 和 `refused`、统计的完整零值结构均视为业务成功状态，不显示为系统故障；
15. SSE 前端实现严格状态机，支持 `metadata`、`token`、`tool_start`、`tool_end`、`message_end`、`error`，流结束前不伪造成功回答；
16. 前端测试不保存或回显真实令牌，使用虚构固定值和 Mock Fetch 验证路由分流、刷新去重、错误解析、SSE 解析和核心组件状态。

## 六、决策理由

1. **高内聚低耦合**：认证状态、HTTP/SSE、路由权限、导航和页面领域分别组织，页面不直接处理令牌刷新与底层协议；
2. **安全边界清晰**：浏览器只访问 FastAPI；前端不出现 Dify Key、Dataset ID、模型密钥或内部工作流配置；
3. **依赖可控**：优先使用浏览器能力和 Vue 官方生态，减少教学项目安装与构建风险；
4. **可演示性**：四个默认工作台与实际角色任务一致，登录后可直接进入最有代表性的流程；
5. **可测试性**：统一路由元数据和请求层便于用 Mock Fetch 验证权限、错误与刷新流程；
6. **可部署性**：相对 API 路径和 Vite 代理便于后续接入 Docker Compose、Nginx、Oracle Cloud 与域名，而不修改页面业务代码。

## 七、影响与风险

- 正面影响：形成独立 Vue3 业务前端；四角色使用同一套基础设施但保持页面边界；可直接支撑答辩演示。
- 当前风险：`sessionStorage` 中的令牌仍可被同源 XSS 读取，因此页面必须避免不可信 HTML 注入；复杂图表使用原生 SVG 会比专业图表库代码更多；第一版页面数量较多，需要按主流程分批验收。
- 回退方案：前端与 FastAPI 通过公开 API 解耦，可移除 `frontend/` 而不影响后端；单个页面可在导航中暂时关闭，不修改接口契约。

## 八、验证计划

- TypeScript 类型检查与 Vite 生产构建；
- Vitest 单元/组件测试：角色默认路由、动态导航、认证恢复、401 刷新与重放、平台错误解析、SSE 事件解析；
- 浏览器冒烟：四角色登录与默认落地页、用户知识问答、客服工单、管理员知识文档、决策者统计；
- 安全检查：构建产物和源码不包含真实 API Key、JWT、Cookie、Dify Dataset ID 或管理员密码；
- 联调失败时记录具体 HTTP 状态、稳定错误码和 `request_id`，不记录用户密码或令牌。

## 十、可用于实习日记的素材

前端阶段的重要工作不是先堆叠页面，而是把认证恢复、令牌刷新、角色路由、错误处理和 SSE 状态机统一起来。通过让四类角色共享应用壳并由同一份角色元数据生成导航和路由守卫，可以减少重复代码，同时理解“前端看不见入口”不等于真正授权，最终权限仍必须由 FastAPI 校验。真实浏览器联调还证明：接口测试通过不等于交互正确，百分比契约、文案承诺与页面入口必须逐项对照验收。
