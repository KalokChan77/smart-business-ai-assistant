# DEC-20260717-021：采用托管文档可见性过滤与分阶段补偿状态

- 日期：2026-07-17
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手、代码审查与验证角色
- 关联任务：知识文档跨租户检索隔离、跨系统失败恢复、分阶段删除、日志脱敏
- 关联实习日记：`project_docs/实习日记/2026-07-17-实习日记.md`

## 一、背景

知识文档上传、索引、重新索引和删除闭环完成后，代码审查发现了三个需要在交付前解决的一致性与隔离问题：第一，Dify Dataset 是共享知识源，如果直接使用全局检索结果，通过 FastAPI 上传的租户文档可能被其他租户命中；第二，Dify 上传或重建已经成功、但本地数据库首次保存失败时，任务可能长期保持 active，甚至留下难以清理的上游文档；第三，Dify 删除成功后，如果本地文件删除失败，本地台账可能继续保存已经失效的 Dify Document ID。

真实冒烟还发现，`httpx` 默认 INFO 请求日志会记录完整上游 URL，而 URL 路径中可能包含 Dataset、Document 或 batch 等内部标识。虽然这些标识不是 API Key，也不应进入普通应用访问日志。

## 二、需要解决的问题

1. 在暂时共用一个 Dify Dataset 的教学部署中，如何让 FastAPI 托管上传文档遵守租户可见性，同时继续保留 5 份预置教学文档的共享能力；
2. PostgreSQL、本地文件系统和 Dify 不能共享同一个事务时，如何避免保存失败造成永久 active job 或无法追踪的上游资源；
3. 删除过程如何记录“Dify 已删除”和“本地文件已删除”两个阶段，使中途失败后可以安全重试；
4. 如何防止第三方 HTTP Client 的默认日志泄露内部 URL 标识；
5. 如何在不引入 Celery、Outbox 或独立 Dataset 的前提下完成本地教学规模的可靠验收。

## 三、可选方案

### 方案 A：每个租户使用独立 Dify Dataset

- 优点：隔离边界最清晰，Dify 检索天然不会返回其他租户文档。
- 缺点：需要为每个租户创建、配置和维护 Dataset 与凭据；不适合当前单机教学阶段，也会改变现有预置共享知识库的使用方式。

### 方案 B：使用 Dify metadata 在检索时过滤租户

- 优点：仍可使用一个 Dataset，租户规则可以下推给检索系统。
- 缺点：需要调整文档写入、metadata 契约和现有 economy 检索流程；对当前 Dify 版本和教学数据的迁移成本较高。

### 方案 C：在 FastAPI 中维护 Dify Document ID 台账并执行可见性过滤

- 内容：Dify 返回召回记录后，FastAPI 根据本地 `knowledge_documents` 台账判断记录是否属于托管文档；托管文档仅对所属租户可见，已软删除文档不可见，未登记的预置文档继续作为平台共享文档。
- 优点：不改变现有 Dataset 和前端 API；可以立即封闭托管文档的跨租户检索风险；规则可使用真实 PostgreSQL 集成测试验证。
- 缺点：检索已经在 Dify 中发生，过滤属于应用层防护；如果未来所有文档都要求严格租户隔离，仍需升级为独立 Dataset 或 metadata 方案。

### 方案 D：引入 Celery、Outbox 和完整补偿任务

- 优点：能够自动发现和处理跨系统不一致，更接近生产级架构。
- 缺点：会增加队列、幂等、重试和运维复杂度，超出当前“本地测试成功”的教学目标。

### 方案 E：记录关键阶段并在同步接口中执行轻量补偿

- 内容：保留当前按需状态刷新方式；上传或重建保存失败时写入可恢复失败态；无 batch 的 active job 超时后自动失败；删除任务分别记录 Dify 删除和文件删除阶段。
- 优点：不新增基础设施，能够把跨系统操作从一次不可见的长事务拆成可审计、可重试步骤。
- 缺点：极端进程崩溃仍可能需要后续清理任务或人工检查。

## 四、最终决策

采用方案 C 与方案 E，并固定以下规则：

1. Dify 中未登记到业务数据库的预置文档继续视为平台共享教学文档；
2. 通过 FastAPI 上传并登记在 `knowledge_documents` 的托管文档，只允许所属租户参与知识回答；已软删除托管文档不再可见；
3. 可见性判断必须在 Dify 检索完成后、生成引用和答案之前执行；数据库不可用时安全失败并返回 `knowledge_visibility_unavailable`，不得直接放行托管记录；
4. Dify 上传或重新索引成功、首次本地保存失败时，将文档和任务恢复为 `failed`，保留必要的内部 Dify Document ID 与 batch 以便删除或重建，但不在公开响应和日志中返回这些字段；
5. 如果恢复状态的第二次保存仍失败，对新上传的 Dify 文档执行 best-effort 补偿删除；
6. pending/processing 任务没有 batch 且持续超过 300 秒时，由任务查询接口将其标记为 `failed`，错误码固定为 `knowledge_document_stale_active_job`，解除 active job 阻塞；
7. 删除顺序固定为：删除 Dify 文档并把 `dify_document_id` 清空、记录 `dify_deleted_at`、删除本地文件并记录 `file_deleted_at`、最后软删除文档和完成任务；
8. 如果本地文件删除失败，已经清空的 Dify Document ID 不得恢复；管理员再次调用 DELETE 时只继续未完成阶段，不重复删除已经不存在的上游文档；
9. `httpx` 与 `httpcore` 日志级别固定为 `WARNING` 且禁止向根 Logger 传播，避免 INFO 请求 URL 进入应用日志；
10. Dataset ID、Dify Document ID、batch、存储绝对路径和任何凭据都属于内部信息，不进入平台响应、决策日志或实习日记。

## 五、决策理由

该方案在当前单机教学规模下，能够用最小新增复杂度解决真实代码审查发现的问题。租户可见性集中在独立 Policy 中，知识服务只依赖统一接口，Dify Retriever 不需要理解业务租户，从而保持高内聚、低耦合。上传、重建和删除的补偿逻辑集中在文档 Service，路由和前端无需了解第三方状态细节。

分阶段状态不能提供跨 PostgreSQL、文件系统和 Dify 的强事务，但可以明确系统已经完成到哪一步，并使失败后的重试行为可验证。对于本科实训项目，这比立即引入 Celery、Outbox 和多租户 Dataset 更符合开发周期、演示范围和维护成本。

## 六、影响与风险

- 正面影响：FastAPI 托管文档实现租户检索隔离；保存失败不再永久占用 active job；删除中断可以幂等恢复；第三方 INFO URL 不再进入应用日志。
- 潜在风险：预置教学文档仍是平台共享；应用层过滤不能替代真正的索引级隔离；连续数据库故障或进程在极窄窗口崩溃时仍可能产生待清理资源；当前日志保护依赖固定 Logger 配置，而不是对 URL path 逐段脱敏。
- 控制措施：真实双租户冒烟、唯一 active job 索引、失败状态与阶段时间戳、best-effort 删除、稳定错误码、全量回归测试和敏感日志检查共同形成防护。
- 回退方案：可关闭托管上传路由并保留预置知识查询；后续若进入正式多租户部署，迁移到独立 Dataset 或 metadata 过滤，并增加后台补偿扫描任务。

## 七、验证证据

- 代码审查提出的跨租户检索、mutation 保存失败和删除中断三个问题均已完成根因修复；复审未发现新的 P1/P2；
- PostgreSQL 集成测试证明：所属租户可见托管文档和共享文档，其他租户只能看到共享文档，托管文档软删除后不再可见；
- 服务层回归测试证明：Dify mutation 成功后首次保存失败会落为失败态、保留内部恢复信息并解除 active job；无 batch 的陈旧任务会自动失败；
- 删除回归测试证明：Dify 删除成功而本地文件失败后，台账不保留陈旧 Dify ID，再次删除不会重复调用上游删除；
- 当前测试集合共收集 184 项；定向审查测试和本轮全量测试全部通过，`compileall`、`pip check`、迁移 head 与 `alembic check` 均通过；
- 真实双租户冒烟通过文档上传、索引完成、非所属租户无匹配、所属租户可回答、重新索引、删除和临时数据清理；
- 清理一次性引导目录后再次执行三类 Dify Service API、知识查询和文档生命周期冒烟，结果仍全部通过；

## 九、可用于实习日记的素材

这次审查让我认识到，共享知识库、业务租户台账和真正的租户隔离是三个不同概念。跨数据库、文件系统和第三方平台的操作也不能依靠一个普通事务解决，必须记录阶段、设计补偿和验证重试。除此之外，第三方库的默认日志也可能暴露内部标识，安全检查不能只关注自己主动编写的日志内容。
