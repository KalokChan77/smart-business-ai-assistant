# DEC-20260718-030：业务栈一键启动与幂等演示账号初始化

- 日期：2026-07-18
- 状态：已接受
- 参与角色：学生开发者、AI 编程助手
- 关联任务：Docker Compose 本地启动、演示账号、根 README、答辩准备
- 关联实习日记：`project_docs/实习日记/2026-07-18-实习日记.md`

## 一、背景

前后端主功能与四角色浏览器冒烟已完成，但新环境仍缺少统一入口：业务 PostgreSQL/Redis、迁移、演示账号、FastAPI 与 Vue 开发服务需要多条命令分别执行。技术方案将 Docker Compose 与 README 列为核心验收，答辩也需要稳定、可重复的演示账号，而不能依赖每次临时创建随机租户。

## 二、需要解决的问题

1. 业务依赖与 Dify 是否合并进同一个 Compose；
2. 演示账号如何在不把密码写入仓库的前提下可重复初始化；
3. 一键脚本应启动哪些进程，停止时是否默认关掉数据库；
4. 根 README 如何同时覆盖安装、启动、验证与安全约定。

## 三、可选方案

### 方案 A：把 Dify 与业务栈合并为单一 Compose

- 优点：一条命令启动全部。
- 缺点：Dify 服务多、日志嘈杂，故障隔离差，与既有“业务库独立于 Dify”决策冲突。

### 方案 B：业务栈一键脚本 + Dify 保持独立 Compose

- 优点：符合已接受的隔离决策；排错清晰；脚本只管理本项目可控部分。
- 缺点：首次仍需单独启动 Dify。

### 方案 C：不做脚本，只写文档命令

- 优点：实现成本最低。
- 缺点：答辩前恢复环境步骤多，容易遗漏迁移或账号。

## 四、最终决策

采用方案 B，并固定：

1. `deploy/app-compose.yml` 仅负责业务 PostgreSQL 与 Redis；
2. `scripts/dev_up.sh` 顺序执行：Compose up → 迁移 → 演示账号 → 启动后端/前端开发进程；
3. `scripts/dev_down.sh` 默认只停前后端进程，数据服务保持运行；`STOP_DATA=1` 时再 stop Compose；
4. 新增独立 `DemoTenantBootstrapper`，在 `users` 领域内封装四角色演示账号的幂等初始化，不把演示规则混入通用 `UserService`；
5. 默认演示租户 UUID 固定为 `a1000000-0000-4000-8000-000000000001`，账号名为 `demo-admin`、`demo-user`、`demo-cs`、`demo-decision`；
6. 密码只来自 `DEMO_PASSWORD` 或交互输入，脚本输出仅打印用户名与角色，不打印密码；
7. 已存在账号默认不改密，除非显式 `--reset-password` / `RESET_DEMO_PASSWORD`；
8. 根目录新增 `README.md`，串联安装、启动、验证、安全约定与文档索引。

## 五、决策理由

1. **高内聚低耦合**：初始化逻辑放在用户领域服务，脚本只做编排；Dify 仍独立；
2. **可复现**：固定租户 ID 与账号名，答辩登录稳定；
3. **安全**：密码不入库文件、不进 README、不进决策日志；
4. **教学友好**：一条脚本覆盖最常见恢复路径，同时允许 `SKIP_DEMO_SEED=1` 跳过。

## 六、影响与风险

- 正面影响：新环境启动路径清晰，答辩准备成本下降；
- 潜在风险：本机若已有占用 8000/5173 的进程，脚本会明确退出并提示处理；演示密码若过弱仍可能被同机其他用户猜测（仅限本地教学）；
- 回退方案：可继续手动执行 Compose、alembic、uvicorn、npm run dev；演示初始化可单独运行 `bootstrap_demo_tenant.py`。
